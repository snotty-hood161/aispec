# Spec 维度 → 开发任务映射表

本文件定义如何将模块级 Spec 的 12 个维度和 Design 产出转化为按域分组的可执行开发任务。

## 使用方式
1. 对每个模块，逐维度扫描 Spec 内容，按下表生成对应任务。
2. 同一维度可能产生多个任务（如数据结构维度按实体数拆分）。
3. Design 产出单独映射客户端任务，与 Spec 维度互补。
4. 并非所有维度都会产生任务 — 若 Spec 中某维度为空或标注"不适用"，则跳过。

---

## 一、Spec 维度 → 域 Agent 映射

| Spec 维度 | 主要执行域 | 阶段 | 任务生成规则 |
|-----------|-----------|------|------------|
| 1. 模块职责与边界 | — | — | 不直接生成任务，作为所有任务的上下文背景 |
| 2. 数据结构 | Database | Phase 1 | 每个实体生成 1 个建表任务；枚举/索引合并到对应实体任务中 |
| 3. API 接口 | 服务端 Agent | Phase 2 | 按功能组（CRUD 一组、业务操作一组）拆分，每组 1 个任务 |
| 4. 业务规则 | 服务端 Agent | Phase 2 | P0 规则合并到对应 API 任务中；复杂规则（涉及多实体/事务）单独 1 个任务 |
| 5. 边界条件 | 服务端 Agent + QA | Phase 2 + 5 | 合并到对应 API/业务规则任务的验收标准中；QA 阶段生成边界测试用例任务 |
| 6. 状态机 | Database + 服务端 | Phase 1 + 2 | 枚举定义合并到建表任务；状态流转逻辑合并到对应 API 任务中 |
| 7. 权限模型 | 服务端 + 客户端 | Phase 2 + 4 | 服务端：权限校验合并到 API 任务中；客户端：权限控制合并到页面任务中 |
| 8. 事件与消息 | 服务端 Agent | Phase 2 | 事件发布合并到触发事件的 API 任务中；消费者逻辑单独 1 个任务 |
| 9. 错误处理 | 服务端 + Collaboration | Phase 2 + 3 | 错误码定义合并到 API 任务中；跨端错误码映射生成 Collaboration 任务 |
| 10. 模块依赖关系 | Coordinator | 规划时 | 不直接生成任务，用于确定任务排序和前置依赖 |
| 11. 测试策略 | QA | Phase 5 | 生成测试策略任务（覆盖单元测试、集成测试、E2E 测试要点） |
| 12. 迁移与兼容性 | Database + DevOps | Phase 1 + 6 | 数据迁移脚本合并到 DB 任务；灰度/功能开关生成 DevOps 任务 |

## 二、Design 产出 → 客户端任务映射

| Design 产出 | 执行域 | 阶段 | 任务生成规则 |
|------------|--------|------|------------|
| 设计 Token（颜色/字体/间距） | 客户端 Agent | Phase 4（首个任务） | 生成 1 个 "设计系统初始化" 任务：配置 Token 到项目 |
| 组件库 | 客户端 Agent | Phase 4 | 新增的自定义组件，每个生成 1 个任务 |
| 页面设计稿 | 客户端 Agent | Phase 4 | 每个页面 1 个任务（复杂页面可拆为多个区块任务） |
| 用户流程图 | 客户端 Agent | Phase 4 | 合并到对应页面任务的验收标准中（页面跳转逻辑） |
| 响应式适配 | 客户端 Agent | Phase 4 | 合并到对应页面任务的验收标准中 |

### 无 Design 产出时的退化策略
若 Design Agent 未参与或产出不可用：
- 客户端任务仅基于 Spec 的 API 接口维度生成（每组 API 对应 1 个页面任务）。
- 所有客户端任务标注 "设计输入：待设计稿补充"。
- 不生成 "设计系统初始化" 和自定义组件任务。

## 三、固定任务（每个项目必须生成）

以下任务不依赖 Spec 维度，每个项目固定生成：

### Phase 3 — 前后端协作
| 任务 | 触发条件 | 说明 |
|------|---------|------|
| API 契约文档生成 | 服务端 API 任务全部完成后 | 使用 `rules/templates/frontend-backend/api-contract-template.md` |
| 联调检查清单 | API 契约完成后 | 使用 `rules/templates/frontend-backend/integration-checklist-template.md` |

### Phase 4.5 — 安全审计
| 任务 | 触发条件 | 说明 |
|------|---------|------|
| 安全威胁建模 | 所有开发任务完成后 | 覆盖认证、授权、数据安全、输入校验 |
| 依赖扫描 | 同上 | 第三方依赖漏洞检测 |

### Phase 5 — 测试
| 任务 | 触发条件 | 说明 |
|------|---------|------|
| 测试策略定义 | 安全审计通过后 | 基于 Spec 的测试策略维度 |
| 核心路径 E2E 测试 | 测试策略定义后 | 覆盖 P0 功能的端到端流程 |

### Phase 6 — 部署
| 任务 | 触发条件 | 说明 |
|------|---------|------|
| CI/CD 流水线配置 | 测试通过后 | 构建、测试、部署自动化 |
| 监控告警配置 | CI/CD 完成后 | 日志、指标、告警规则 |

## 四、任务 ID 编码规则

格式：`{MODULE}-{PHASE}-{SEQ}`

| 片段 | 说明 | 示例 |
|------|------|------|
| MODULE | 模块缩写（大写，≤8 字符） | USER、ORDER、PAY |
| PHASE | 阶段缩写 | DB、SRV、COLLAB、UI、SEC、QA、OPS |
| SEQ | 三位序号 | 001、002 |

示例：
- `ORDER-DB-001` — 订单模块：创建 orders 表
- `ORDER-SRV-001` — 订单模块：实现订单 CRUD API
- `ORDER-SRV-002` — 订单模块：实现订单状态流转逻辑
- `ORDER-UI-001` — 订单模块：订单列表页
- `ORDER-UI-002` — 订单模块：订单详情页
- `GLOBAL-COLLAB-001` — 全局：API 契约文档生成
- `GLOBAL-SEC-001` — 全局：安全威胁建模
- `GLOBAL-QA-001` — 全局：测试策略定义

## 五、任务粒度指南

### 粒度过粗的信号（需要拆分）
- 任务涉及 3 个以上实体的 CRUD。
- 任务同时包含前端和后端工作。
- 预估复杂度为 L 且没有明确的完成边界。

### 粒度过细的信号（需要合并）
- 多个任务操作同一个实体的不同字段。
- 任务之间有强顺序依赖且无法独立验收。
- 单个任务不足以产生可验证的产出物。

### 复杂度估算
| 等级 | 定义 | 典型场景 |
|------|------|---------|
| S | 单实体 CRUD / 单页面 / 简单配置 | 创建一个基础表、实现一个列表页 |
| M | 多实体关联 / 业务规则 / 复杂页面 | 订单状态机、带筛选的管理页面 |
| L | 跨模块交互 / 分布式事务 / 复杂流程 | 支付回调处理、多步骤审批流 |

## 六、前置依赖推导规则

| 规则 | 说明 |
|------|------|
| DB 任务无前置依赖（同模块内） | 建表是模块的起点 |
| SRV 任务依赖同模块的 DB 任务 | API 实现需要表结构 |
| COLLAB 任务依赖所有 SRV 任务 | 契约对齐需要 API 全部就绪 |
| UI 任务依赖同模块的 SRV 任务 + COLLAB | 页面需要 API 和契约 |
| UI 任务依赖 Design 产出 | 页面实现需要设计稿 |
| 跨模块 SRV 依赖被调用模块的 SRV 任务 | A 调 B 的接口，B 须先完成 |
| SEC/QA/OPS 依赖所有开发任务 | 质量保障在开发完成后执行 |
