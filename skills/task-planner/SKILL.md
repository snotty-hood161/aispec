# 任务规划器

将模块级 Spec 和 UI 设计稿拆解为可执行的开发任务清单，确保每个任务有明确的执行域、前置依赖和验收标准。

## 何时使用
1. Spec Agent（Phase 0）和 Design Agent（Phase 0.5）均已完成产出后，Coordinator 在制定执行计划时调用。
2. 用户手动要求 "把这个 Spec 拆成开发任务" 时直接调用。
3. 已有 Spec 文档但需要重新规划任务顺序时。

## 执行原则
1. **先读后拆** — 必须先完整阅读 Spec 和 Design 产出，再进行任务拆解。
2. **Spec 定义做什么，Design 定义长什么样** — 两者互补，缺一不可（Design 可缺省，Spec 不可缺省）。
3. **按域分组** — 每个任务明确归属到一个域 Agent。
4. **按阶段排序** — 遵循 `agents/protocols/coordination.md` 的阶段顺序。
5. **依赖显式化** — 每个任务的前置依赖必须标注，不允许隐式假设。
6. **粒度适中** — 单个任务应在一次 AI 执行中可完成（通常对应一个表/一组 API/一个页面）。

## 输入要求

### 必须输入
- **项目级 Spec**：技术选型、架构模式、全局约束。
- **模块级 Spec**（至少一个）：数据结构、API 接口、业务规则、状态机等 12 维度。

### 可选输入
- **UI 设计稿**：页面清单、页面间跳转关系、组件清单、设计 Token。若无 Design 产出，则客户端任务仅基于 Spec 中的 API 和功能描述生成。
- **PRD**：用户故事、验收标准。用于生成验收标准更精准的任务描述。

## 标准工作流（必须执行）

### 1. 收集输入
- 读取 Spec Agent 的项目级 Spec 和所有模块级 Spec。
- 读取 Design Agent 的设计交付文档（如有）：页面清单、组件清单、设计 Token。
- 读取 PRD 中的验收标准（如有）。

### 2. Spec × Design 交叉对齐检查（MUST）

Spec 和 Design 并行产出，必须在拆解任务前检查二者的一致性。按以下清单逐项检查，发现问题记入 **对齐问题清单**：

#### 2.1 页面覆盖度检查
- Spec 中每个模块的 API 接口 → Design 中是否有对应的页面承载这些 API 的调用？
- Design 中每个页面 → Spec 中是否有对应模块的 API 支撑这个页面的数据和操作？
- 不一致类型：
  - **Spec 有但 Design 缺**（如 Spec 定义了"导出报表 API"但 Design 没有导出入口）→ 标记为 `DESIGN-GAP`。
  - **Design 有但 Spec 缺**（如 Design 画了"实时搜索"但 Spec 只定义了分页列表 API）→ 标记为 `SPEC-GAP`。

#### 2.2 数据字段一致性检查
- Design 页面上展示的字段 → Spec 的 API 响应结构中是否包含这些字段？
- Design 表单中的输入字段 → Spec 的 API 请求参数中是否包含？
- 不一致类型：
  - **Design 展示但 API 不返回** → 标记为 `SPEC-GAP`。
  - **API 返回但 Design 不展示**（非敏感字段）→ 标记为 `INFO`（信息级，不阻塞）。

#### 2.3 交互行为一致性检查
- Design 定义的用户操作流程 → Spec 的状态机和业务规则是否支撑？
- Design 定义的错误提示文案 → Spec 的错误码是否覆盖？
- 不一致类型：
  - **Design 流程但 Spec 无状态支撑** → 标记为 `SPEC-GAP`。
  - **Spec 有状态但 Design 无对应交互** → 标记为 `DESIGN-GAP`。

#### 2.4 对齐问题处理策略

| 问题类型 | 严重性 | 处理方式 |
|---------|--------|---------|
| `SPEC-GAP` | 阻塞 | 暂停拆解该模块的 SRV/DB 任务，上报 Coordinator 要求 Spec Agent 补充 |
| `DESIGN-GAP` | 非阻塞 | 照常拆解 DB 和 SRV 任务，客户端任务标注 "待设计稿补充：{缺失页面/组件}"，生成一条反馈给 Design Agent 的补充请求 |
| `INFO` | 不阻塞 | 记录在任务清单的 "备注" 中，不影响拆解 |

#### 2.5 无 Design 产出时
若 Design Agent 未参与或产出不可用，跳过本步骤的 Design 侧检查：
- 所有 DB 和 SRV 任务正常基于 Spec 生成。
- 所有客户端任务标注 "设计输入：待设计稿补充"。
- 输出任务清单时附注 "客户端任务待 Design 产出后细化"。

### 3. 模块排序
- 根据 Spec 中的模块依赖图确定模块的处理顺序（被依赖的模块优先）。
- 检查循环依赖：存在则标注为阻塞问题，上报用户或 Coordinator。

### 4. 按模块拆解任务
对每个模块，按 `references/task-decomposition-map.md` 的映射规则，将 Spec 的 12 个维度转化为具体任务。

每个任务包含：
```
- 任务 ID：{MODULE}-{PHASE}-{SEQ}（如 ORDER-DB-001）
- 标题：{一句话描述}
- 所属模块：{模块名}
- 执行域：{域 Agent 名称}
- 执行阶段：{Phase 1~6}
- 前置依赖：{依赖的任务 ID 列表，无则填"无"}
- Spec 输入：{需要从 Spec 中截取的章节}
- Design 输入：{需要从设计稿中截取的页面/组件，无则填"无"}
- 验收标准：{怎样算完成}
- 预估复杂度：{S / M / L}
```

### 5. 跨模块依赖对齐
- 检查模块 A 的任务是否引用了模块 B 的产出（如模块 A 的 API 调用模块 B 的接口）。
- 确保被引用模块的对应任务排在前面。
- 标注跨模块的集成验证任务。

### 6. 生成任务清单
按以下结构输出完整的任务清单：

```
## 任务清单 — {项目名称}

### 执行概览
| 阶段 | 任务数 | 涉及模块 |
|------|--------|---------|

### Phase 1 — 数据库（Database Agent）
| 任务 ID | 标题 | 模块 | 前置依赖 | 复杂度 |
|---------|------|------|---------|--------|

### Phase 2 — 服务端（{Server} Agent）
（同上结构）

### Phase 3 — 前后端协作（Collaboration Agent）
（同上结构）

### Phase 4 — 客户端（{Client} Agent）
（同上结构）

### Phase 4.5 — 安全审计（Security Agent）
（同上结构）

### Phase 5 — 测试（QA Agent）
（同上结构）

### Phase 6 — 部署（DevOps Agent）
（同上结构）

### Spec × Design 对齐问题
| 问题 ID | 类型 | 模块 | 描述 | 影响的任务 | 处理状态 |
|---------|------|------|------|-----------|---------|

### 关键路径
{标注决定项目最短工期的任务链}

### 并行机会
{标注可并行执行的任务组}
```

### 7. 输出约束
1. 所有输出使用中文。
2. 任务 ID 使用大写英文，模块缩写不超过 8 个字符。
3. 每个任务的验收标准必须可验证（不使用 "合理"、"良好" 等模糊表述）。
4. 若 Design 产出不可用，客户端阶段的任务标注 "待设计稿补充"。
5. 按 `agents/protocols/execution-trace.md` 格式，在输出末尾附执行追溯（skill 名称、任务类型、加载规则清单、跨域联动）。
