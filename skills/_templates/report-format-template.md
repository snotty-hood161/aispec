# 审查报告输出格式（通用模板）

每次执行代码审查后，输出按以下固定结构组织。域 SKILL.md 中的 `context_type` 决定"审查概要"的上下文字段。

---

## 1. 审查概要

- **上下文**：{按域 SKILL.md 的 context_type 填写，如部署模式 / 应用类型 / UI 框架 / 目标平台}
- **涉及模块**：列出命中的业务模块
- **变更文件数**：N 个
- **命中规则数**：N 条

## 2. 变更文件清单

| 文件路径 | 变更类型 | 命中检查项 |
|----------|----------|------------|
| `path/to/file` | 修改 | XX-01, XX-02 |

## 3. P0 阻断项

> 以下任一未通过则结论为 `Request Changes`。

| 检查项 ID | 检查项描述 | 违规文件 | 违规详情 | 修复建议 |
|-----------|-----------|----------|----------|----------|

## 4. P1 建议项

> 以下未通过允许带条件合并，但必须登记技术债。

| 检查项 ID | 检查项描述 | 违规文件 | 违规详情 | 改进建议 |
|-----------|-----------|----------|----------|----------|

## 5. 通过项汇总

| 检查项 ID | 检查项描述 | 结果 |
|-----------|-----------|------|

## 6. 不适用项

| 检查项 ID | 原因 |
|-----------|------|

## 7. 本次实际读取文件清单

列出本次审查中实际打开的规则文件路径，说明为什么不需要读取其余规则文件。

## 8. 审查结论

- **P0 阻断项**：N 项通过 / M 项未通过
- **P1 建议项**：N 项通过 / M 项未通过
- **结论**：`Approve` / `Request Changes` / `Conditional Approve`

### 结论判定规则
- `Approve`：全部 P0 通过，P1 全部通过或无 P1 项。
- `Request Changes`：存在任一 P0 未通过。
- `Conditional Approve`：P0 全部通过，存在 P1 未通过且已登记技术债与回收计划。

## 9. 修复建议（仅 Request Changes 时）

按优先级列出修复建议，最多 5 条：

1. [P0] 修复描述 → 具体操作
2. [P0] 修复描述 → 具体操作
3. [P1] 改进描述 → 具体操作

## 10. 执行追溯

按 `agents/protocols/execution-trace.md` 格式附执行追溯摘要（调用 Skill、任务类型、加载规则、跨域规则、跨域联动）。
