# 规则维护参考资料合集（通用模板）

本文件合并了所有 `*-rules-maintainer` 共用的 checklist、output-contract、change-modes 三份参考资料。

---

## 一、执行核对清单

### P0 阻塞项（必须通过）
1. 结构校验通过：域 SKILL.md 声明的 `validate_script`。
2. 语义校验通过：域 SKILL.md 声明的 `lint_script`。
3. 新增/变更规则已按"单一来源"落位，无跨文件重复正文。
4. `index.md` 已同步更新，且所有规则文件可被索引到。
5. 若存在规则冲突，已给出冲突点、影响面、备选方案和最终决策。

### P1 改进项（可条件通过）
1. 规则措辞已尽量统一为"必须/禁止/应当"。
2. 目录与示例保持最新，避免陈旧样例误导实现。
3. 输出摘要包含"后续建议"和"潜在扩展项"。

### 交付输出核对
1. 已列出变更文件清单。
2. 已列出命中规则文件与原因（需求 -> 规则映射）。
3. 已附两个脚本校验输出。
4. 已给出 P0/P1 风险结论（Approve / Request Changes / Conditional Approve）。

---

## 二、输出协议（必须遵循）

每次执行本技能后，输出按以下固定结构组织。

### 1. 本次目标
- 用 1-2 句复述用户目标与本次模式（新增/重构/审计/冲突修复）。

### 2. 需求 -> 规则映射
- 列出命中规则文件（主定义文件在前）。
- 每个文件给出命中理由（对应用户需求点）。

### 3. 变更清单
- 列出改动文件路径。
- 每个文件说明"新增/修改/删除"的关键点。

### 4. 校验结果
- 结构校验：`validate_script` 结果（通过/失败 + 摘要）。
- 语义校验：`lint_script` 结果（通过/失败 + 摘要）。
- 如失败，列出阻塞原因和待修复项。

### 5. 风险分级结论
- `P0`：阻塞项是否全部通过。
- `P1`：建议改进项与技术债登记建议。
- 给出结论：`Approve` / `Request Changes` / `Conditional Approve`。

### 6. 冲突与决策记录
- 如有冲突，列出冲突规则、取舍方案、最终决策与理由。
- 无冲突时明确写"无冲突"。

### 7. 下一步建议
- 仅给 1-3 条可执行建议。

## 三、变更模式（执行模板）

按需求类型选择模式，确保流程稳定且可复用。

### 模式 A：新增规则
1. 先用映射文件选唯一主定义文件。
2. 在主文件新增规则条目，使用可验证措辞（必须/禁止/应当）。
3. 如需跨文件协同，仅在关联文件补"引用性约束"，不复制正文。
4. 若新增文件，更新 `index.md`。
5. 运行双校验脚本并输出结果。

### 模式 B：规则重构/迁移
1. 识别重复与冲突条目，确定保留文件（单一来源）。
2. 将重复正文迁移到主文件，其他文件改为引用或删减。
3. 保证迁移后语义不弱化（尤其 P0 阻塞项）。
4. 执行双校验并汇报前后差异。

### 模式 C：一致性审计
1. 不先改文档，先运行双校验收集问题。
2. 按"结构问题 -> 语义问题 -> 风险分级"排序处理。
3. 逐项修复后再次全量校验。
4. 输出"发现问题清单 + 修复清单 + 残留风险"。

### 模式 D：冲突修复
1. 标注冲突来源（用户诉求、common、profile、数据库规则）。
2. 给出至少两个备选方案（推荐方案 + 保守方案）。
3. 明确影响范围（哪些文件、哪些团队、哪些约束）。
4. 用户确认后实施并记录最终决策理由。

### 模式 E：仅更新索引或结构
1. 保持规则正文不变，仅修复索引、路径、结构漂移。
2. 确保 `index.md` 完整、唯一、无失效路径。
3. 输出"结构修复说明"，避免误导为语义变更。
