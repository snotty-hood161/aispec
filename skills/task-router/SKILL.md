# 任务域路由器

接收完整业务任务描述，分析涉及的技术域，按正确顺序调度各域 coding-guide。

## 何时使用
1. 用户提出跨域业务任务（如"做一个订单管理功能"涉及数据库 + 后端 + 前端）。
2. 用户提出单域但未明确指定域的任务（如"加一个导出功能"需要判断是前端还是后端）。
3. 用户手动指定 `$task-router` 时。

## 执行原则
1. 以 `references/domain-detect-map.md` 为域识别依据。
2. 以 `rules/index.md` 为跨域冲突仲裁依据。
3. 每个域的具体编码规范由对应的 `*-coding-guide` 负责加载，本 skill 不直接读取域内规则文件。
4. 禁止一次性加载所有域的规则。

## 标准工作流（必须执行）

### 1. 任务分析
- 提取任务中的技术关键词（API、数据库、页面、组件、定时任务...）。
- 用 `references/domain-detect-map.md` 映射到涉及的域。
- 若无法确定，询问用户确认。

### 2. 确定执行顺序
- 默认顺序：数据库 → 服务端（Go/.NET）→ 前后端协作 → 客户端（前端/桌面/移动端）。
- 可并行的域标注为并行（如 iOS 和 Android 可并行）。
- 数据库变更必须先于服务端；服务端必须先于或同步于客户端。

### 3. 逐域调度
- 按顺序调度各域的 `*-coding-guide`。
- 每个 coding-guide 自行识别编码场景并加载对应规则。
- 跨域契约（如 API 接口定义）在进入客户端域前，先由 `$frontend-backend-coding-guide` 处理。

### 4. 输出任务摘要
- 列出涉及的域与执行顺序。
- 列出每个域实际加载的规则文件清单。
- 标注跨域依赖关系。
- 按 `agents/protocols/execution-trace.md` 格式，在输出末尾附执行追溯（skill 名称、任务类型、加载规则清单、跨域联动）。
