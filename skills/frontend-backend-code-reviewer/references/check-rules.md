# 可自动检查的规则条目清单

本文件列出可在代码审查中自动/半自动检查的规则条目，按来源章节分组。

## 使用方式
1. 根据变更文件路径与内容，匹配下方检查项。
2. 每项标注检查方式：`契约比对`（前后端定义一致性）/ `模式匹配`（代码中的关键模式）/ `人工审查`。
3. P0 项必须全部通过，P1 项允许带条件通过。

---

## 一、协作原则

| ID | 优先级 | 检查项 | 检查方式 | 说明 |
|----|--------|--------|----------|------|
| CO-01 | P0 | 接口契约确认后再并行开发 | 人工审查 | 检查是否存在对应的 API 契约文档；无契约文档的接口变更视为违规 |
| CO-02 | P0 | 兼容窗口内不破坏旧调用方 | 契约比对 | 非兼容变更必须附带迁移计划，且在兼容窗口期内保留旧接口 |
| CO-03 | P0 | 联调通过后再灰度发布 | 人工审查 | 检查联调检查清单（integration-checklist）是否已完成并签字 |

## 二、API 契约规范

| ID | 优先级 | 检查项 | 检查方式 | 说明 |
|----|--------|--------|----------|------|
| AC-01 | P0 | 接口包含完整定义 | 契约比对 | 契约文档须包含：请求参数、响应结构、错误码、鉴权方式、幂等标识 |
| AC-02 | P0 | 枚举值由服务端定义为单一来源 | 模式匹配 | 前端不得自定义与服务端重复的枚举常量，须从 API 响应或共享定义消费 |
| AC-03 | P0 | 新增字段默认可选+向后兼容 | 契约比对 | 新增字段须标记为 optional，不得将已有字段从 optional 改为 required |
| AC-04 | P0 | 删除字段先完成使用方清理 | 模式匹配 | 删除字段前须确认前端已不引用该字段，或标记为 deprecated 并设定清理期限 |
| AC-05 | P0 | 时间字段明确时区与格式 | 模式匹配 | 时间字段须标注格式（如 ISO 8601）与时区（如 UTC），前后端保持一致 |

## 三、错误码与提示协同

| ID | 优先级 | 检查项 | 检查方式 | 说明 |
|----|--------|--------|----------|------|
| EC-01 | P0 | 错误码语义前后端一致 | 契约比对 | 前端处理的错误码须与服务端返回的错误码定义一致，不得自行映射语义 |
| EC-02 | P0 | 业务错误与系统错误区分 | 模式匹配 | 服务端须区分业务错误（可展示给用户）和系统错误（仅日志），前端据此决定展示策略 |
| EC-03 | P0 | 高风险操作失败有可操作提示 | 模式匹配 | 支付、删除等高风险操作失败时，前端须展示包含操作指引的提示，而非通用错误信息 |

## 四、变更分级与流程

| ID | 优先级 | 检查项 | 检查方式 | 说明 |
|----|--------|--------|----------|------|
| CG-01 | P0 | 非兼容变更有评审与迁移计划 | 人工审查 | 非兼容变更须经跨端评审，并附带书面迁移计划 |
| CG-02 | P0 | 非兼容变更包含影响面、迁移步骤、回滚方案、完成时间 | 人工审查 | 迁移计划须包含四要素，缺一视为不完整 |
| CG-03 | P0 | 兼容变更标注为可并行开发 | 契约比对 | 兼容变更在契约文档中明确标注，允许前后端并行开发 |

## 五、联调与测试

| ID | 优先级 | 检查项 | 检查方式 | 说明 |
|----|--------|--------|----------|------|
| IT-01 | P0 | 联调前完成契约自测 | 人工审查 | 前后端各自按契约完成自测，联调前提供自测报告 |
| IT-02 | P0 | 联调覆盖成功路径+鉴权失败+参数错误+幂等重试 | 人工审查 | 联调用例须覆盖正常流程与异常场景，至少包含四类场景 |
| IT-03 | P0 | 提测前通过前端 lint+typecheck+test 与服务端质量门禁 | 模式匹配 | CI 流水线须包含对应的质量检查步骤，且状态为通过 |

## 六、发布与回滚

| ID | 优先级 | 检查项 | 检查方式 | 说明 |
|----|--------|--------|----------|------|
| DR-01 | P0 | 接口变更采用灰度发布 | 人工审查 | 涉及接口变更的发布须采用灰度策略，不得全量直接发布 |
| DR-02 | P0 | 默认发布顺序：先服务端兼容发布，再前端发布切换 | 人工审查 | 确保服务端先上线兼容版本，前端再切换到新接口 |
| DR-03 | P0 | 回滚策略可执行 | 人工审查 | 发布记录中须包含回滚步骤，且经过验证可实际执行 |

## 七、观测与问题定位

| ID | 优先级 | 检查项 | 检查方式 | 说明 |
|----|--------|--------|----------|------|
| OB-01 | P1 | 前后端使用一致的 traceId/requestId 透传 | 模式匹配 | 前端请求头须包含 traceId，服务端须在日志中记录相同 traceId |
| OB-02 | P1 | 核心接口监控包含成功率、耗时、错误码分布 | 人工审查 | 核心接口须配置监控面板，覆盖三项指标 |
| OB-03 | P1 | 线上故障按"契约→网络→服务端日志→前端映射"排查 | 人工审查 | 故障排查流程须遵循标准顺序，避免跳步定位 |
