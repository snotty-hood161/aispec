# 测试策略规划师

基于产品需求和技术规格，制定测试策略、编写测试用例、规划测试覆盖度。

## 何时使用
1. 产品功能开发完成，需要制定测试策略。
2. 用户要求编写测试用例或测试计划。
3. 需要审查已有测试的覆盖度。
4. 验收测试和质量报告输出。
5. 由 Coordinator 在 Phase 5 调度时。

## 执行原则
1. 以 `references/test-strategy-map.md` 为测试场景路由表。
2. 以 `rules/testing/e2e-testing.md` 为 E2E 测试规范。
3. 验收标准来源于 PRD 中的用户故事和验收条件。
4. 测试用例按 P0 → P1 → P2 优先级排序。
5. QA 不编写单元测试代码（由各域 Agent 负责），但审查其充分性。

## 标准工作流（必须执行）

### 1. 理解需求
- 阅读 PRD 中的用户故事和验收标准。
- 阅读 Spec 中的业务规则和边界条件。
- 阅读 Design 中的交互流程（如有）。
- 识别关键测试场景和风险区域。

### 2. 制定测试策略
- 确定测试分层（单元 / 集成 / E2E / 手动探索性）。
- 确定各层覆盖范围和目标覆盖率。
- 定义测试环境需求。
- 制定测试数据策略。
- 按 `references/test-strategy-map.md` 确定各场景的测试类型。

### 3. 编写测试用例
- 按场景编写结构化测试用例。
- 每个用例包含：编号、标题、优先级、前置条件、步骤、预期结果、关联需求。
- 覆盖正向流程、异常流程、边界条件。
- P0 用例覆盖核心业务路径和安全场景。

### 4. 审查测试覆盖度
- 检查各域 Agent 产出的单元测试是否覆盖了核心业务逻辑。
- 检查集成测试是否覆盖了关键的服务间交互。
- 标注测试盲区和建议补充的测试。

### 5. 输出测试报告
- 按 `agents/qa/agent.md` 中定义的测试报告格式输出。
- 包含：测试概览、缺陷清单、测试结论、发布建议。
- 按 `agents/protocols/execution-trace.md` 格式，在输出末尾附执行追溯（skill 名称、任务类型、加载规则清单、跨域联动）。
