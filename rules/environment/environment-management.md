# 环境管理与配置规范

## Skill 协作
1. DevOps 任务优先使用 `$devops-engineer`，自动加载本规则。
2. 各域 coding-guide 中的 configuration.md / env-config.md 为域内配置细则，本文件为跨域环境管理基线。
3. 跨域业务任务使用 `$task-router` 自动路由。

## 环境分级（MUST）

| 环境 | 代号 | 用途 | 访问范围 | 数据策略 | 稳定性要求 |
|------|------|------|---------|---------|-----------|
| 开发 | dev | 开发调试 | 开发团队内网 | 模拟/测试数据 | 允许频繁重启 |
| 测试 | test | 自动化测试 | CI/CD 系统 | 自动重置的测试数据 | 测试期间稳定 |
| 预发 | staging | 上线前验证 | 受限人员 | 脱敏生产数据快照 | 与生产一致 |
| 生产 | prod | 线上服务 | 公网/客户 | 真实数据 | 高可用 |

### 环境一致性原则
1. staging 的技术栈、配置结构、基础设施必须与 prod 保持一致。
2. 环境间差异仅限于：连接地址、密钥、日志级别、功能开关。
3. 禁止在 prod 上存在但 staging 上不存在的配置项。

## 配置管理（MUST）

### 配置分层

```
配置优先级（从高到低）：
1. 运行时覆盖（环境变量 / 命令行参数）
2. 环境级配置（配置中心 / 环境特定配置文件）
3. 应用默认值（代码中的 fallback 值）
```

### 配置文件规范
1. 配置文件必须按环境分离：`config.dev.yaml`、`config.staging.yaml`、`config.prod.yaml`。
2. 公共配置抽取到 `config.base.yaml`，环境配置只覆盖差异项。
3. 配置文件模板（不含敏感值）提交到版本库：`config.example.yaml`。
4. 禁止将含有真实密钥的配置文件提交到版本库。

### 环境变量命名规范
1. 统一大写，使用下划线分隔：`APP_DATABASE_HOST`。
2. 使用统一前缀区分应用：`MYAPP_`。
3. 命名结构：`{APP}_{MODULE}_{KEY}`。
4. 布尔值使用 `true` / `false`，不使用 `1` / `0` / `yes` / `no`。

| 类别 | 命名模式 | 示例 |
|------|---------|------|
| 数据库 | `{APP}_DB_{KEY}` | `MYAPP_DB_HOST`、`MYAPP_DB_PORT` |
| 缓存 | `{APP}_REDIS_{KEY}` | `MYAPP_REDIS_HOST`、`MYAPP_REDIS_PASSWORD` |
| 消息队列 | `{APP}_MQ_{KEY}` | `MYAPP_MQ_BROKER_URL` |
| 第三方服务 | `{APP}_{SERVICE}_{KEY}` | `MYAPP_OSS_ACCESS_KEY` |
| 应用配置 | `{APP}_{KEY}` | `MYAPP_LOG_LEVEL`、`MYAPP_PORT` |

## 密钥管理（MUST）

### 密钥存储
1. 开发环境：`.env` 文件（已加入 .gitignore）。
2. CI/CD 环境：平台内置 Secrets 管理（GitHub Secrets / GitLab CI Variables）。
3. 生产环境：密钥管理服务（HashiCorp Vault / AWS KMS / Azure Key Vault）或容器运行时密钥注入。

### 密钥轮换
1. 数据库密码、API Key 轮换周期 ≤ 90 天。
2. TLS 证书到期前 30 天自动续期。
3. 密钥轮换必须支持双密钥并存期（新旧密钥同时有效的过渡窗口）。
4. 轮换后旧密钥在过渡窗口结束后立即失效。

## 配置中心（SHOULD）

### 适用场景
- 微服务架构（5+ 服务实例）推荐使用配置中心。
- 单体应用可使用环境变量 + 配置文件。

### 配置中心要求
1. 支持按环境隔离的命名空间。
2. 配置变更必须有审计日志。
3. 支持配置回滚。
4. 支持灰度发布配置（按比例推送）。
5. 敏感配置必须加密存储。

## 功能开关（SHOULD）

### 开关类型
| 类型 | 用途 | 生命周期 | 示例 |
|------|------|---------|------|
| 发布开关 | 控制未完成功能的可见性 | 功能发布后移除 | `FEATURE_NEW_CHECKOUT` |
| 实验开关 | A/B 测试 | 实验结束后移除 | `EXP_NEW_HOMEPAGE_V2` |
| 运维开关 | 动态降级/限流 | 长期保留 | `OPS_ENABLE_CACHE` |
| 权限开关 | 按用户/租户开放功能 | 长期保留 | `PERM_ADVANCED_ANALYTICS` |

### 开关管理规范
1. 开关命名：`{TYPE}_{FEATURE_NAME}`，全大写下划线分隔。
2. 每个开关必须记录：创建日期、负责人、预期移除日期、默认值。
3. 发布开关和实验开关必须在功能稳定后 30 天内清理。
4. 开关数量告警：单个服务 > 20 个活跃开关时触发清理提醒。

## 环境初始化检查清单（MUST）

新环境搭建时必须完成以下检查：

```
## 环境初始化检查清单

### 基础设施
- [ ] 网络隔离配置（VPC / 安全组）
- [ ] DNS 解析配置
- [ ] TLS 证书安装
- [ ] 负载均衡配置

### 数据层
- [ ] 数据库实例创建与初始化（schema.sql）
- [ ] 缓存实例配置
- [ ] 消息队列配置
- [ ] 对象存储 Bucket 创建

### 应用层
- [ ] 环境变量 / 配置文件部署
- [ ] 密钥注入验证
- [ ] 健康检查端点可达
- [ ] 日志收集配置

### 安全
- [ ] 防火墙规则配置
- [ ] WAF 规则（如适用）
- [ ] 密钥轮换策略配置
- [ ] 审计日志开启

### 监控
- [ ] 指标采集配置
- [ ] 日志聚合配置
- [ ] 告警规则配置
- [ ] 告警通知渠道验证
```
