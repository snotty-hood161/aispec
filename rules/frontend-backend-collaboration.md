# rules/frontend-backend-collaboration.md

## 目的
1. 统一前端与服务端协作方式，降低联调返工与线上回归风险。
2. 明确“谁定义、谁实现、谁验证、谁兜底”的跨端责任边界。

## 适用范围
1. 适用于所有涉及前后端接口、鉴权、联调、发布协同的需求。
2. 与 `rules/frontend/`、`rules/flutter/`、`rules/go-server/`、`rules/dotnet-server/`、`rules/database/database.md` 联合生效。

## 协作原则（MUST）
1. 先契约后开发：接口契约（路径、方法、字段、错误码）确认后再并行开发。
2. 先兼容后替换：线上字段和行为变更必须走兼容窗口，不得直接破坏旧调用方。
3. 先验证后发布：联调通过 + 回归通过后再灰度发布。
检查方式：人工审查 + CI 阻断  
阻断级别：阻断合并

## API 契约规范（MUST）
1. 接口必须包含：请求参数、响应结构、错误码、鉴权方式、幂等要求。
2. 枚举值由服务端定义为单一来源，前端仅消费，不得自创枚举语义。
3. 新增字段默认“可选 + 向后兼容”；删除字段必须先完成使用方清理。
4. 时间字段必须明确时区与格式（建议 ISO 8601 或时间戳）。
检查方式：人工审查  
阻断级别：阻断合并

## 错误码与提示协同（MUST）
1. 服务端统一错误码语义，前端统一错误码映射与用户提示。
2. 业务错误与系统错误必须区分，不得全部返回同一错误语义。
3. 高风险操作失败必须给出可操作提示（重试/联系支持/回退动作）。
检查方式：人工审查 + 静态扫描  
阻断级别：阻断合并

## 变更分级与流程（MUST）
1. `兼容变更`：新增字段、扩展枚举、新增接口；可并行开发。
2. `非兼容变更`：删除字段、语义变更、鉴权变更；必须先评审并出迁移计划。
3. 所有非兼容变更必须包含：影响面、迁移步骤、回滚方案、完成时间。
检查方式：人工审查  
阻断级别：阻断合并

## 联调与测试（MUST）
1. 联调前必须完成契约自测：请求示例、响应示例、错误示例。
2. 联调阶段至少覆盖：成功路径、鉴权失败、参数错误、幂等重试。
3. 提测前必须通过：前端 `lint + typecheck + test`，服务端对应质量门禁。
检查方式：CI 阻断 + 人工审查  
阻断级别：阻断合并

## 发布与回滚（MUST）
1. 涉及接口变更的需求必须采用灰度发布策略。
2. 发布顺序默认：先服务端兼容发布，再前端发布切换。
3. 回滚策略必须可执行：服务端可回滚版本、前端可回退开关或版本。
检查方式：人工审查  
阻断级别：阻断合并

## 观测与问题定位（SHOULD）
1. 前后端使用一致的 `traceId/requestId` 透传策略。
2. 核心接口监控至少包含成功率、耗时、错误码分布。
3. 线上故障排查按“契约 -> 网络 -> 服务端日志 -> 前端映射”顺序执行。
检查方式：人工审查  
阻断级别：告警记录

## 冲突优先级
1. 数据库结构与迁移相关冲突，以 `rules/database/database.md` 为准。
2. 服务端实现细节冲突，以对应服务端规则域（`rules/go-server/` 或 `rules/dotnet-server/`）为准。
3. 前端实现细节冲突，以 `rules/frontend/` 为准。
4. 跨端协作冲突无法消解时，以“更严格且可验证”规则为准。

## Skill 协作
1. 编写涉及前后端交互的代码时使用 `$frontend-backend-coding-guide` 自动加载对应规则。
2. 跨域业务任务使用 `$task-router` 自动路由到涉及的域。
3. 规则维护使用 `$frontend-backend-rules-maintainer`。

## 配套模板（MUST）
1. 接口契约必须使用：`rules/templates/frontend-backend/api-contract-template.md`。
2. 联调提测必须附：`rules/templates/frontend-backend/integration-checklist-template.md`。
3. 发布变更必须附：`rules/templates/frontend-backend/release-rollback-record-template.md`。
