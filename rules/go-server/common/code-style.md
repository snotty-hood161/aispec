# rules/go-server/common/code-style.md

## 命名与组织
1. 包名使用小写短词，禁止下划线与无意义缩写。
2. 导出类型和函数必须有注释，注释说明职责、输入约束和副作用。
3. 构造函数统一命名 `NewXxx`，依赖必须通过构造函数注入。
4. 接口遵循”谁使用谁定义”，避免预建大而全接口。
5. 文件命名必须体现职责，禁止 `util.go`、`common.go`、`misc.go` 等模糊命名承载多责任逻辑。

## 注释规范

### MUST
1. 注释语言统一使用中文；与外部开源库交互的接口适配文件允许使用英文。
2. 每个源文件顶部必须包含模块用途说明注释（一句话概括文件职责）。
3. 所有导出的函数、类型、接口、常量必须有中文注释，说明”做什么”和”为什么”。格式遵循 Go 标准：`// FunctionName 中文说明`。
4. 非导出但逻辑复杂的函数（超过 30 行或包含分支、循环、并发）必须在函数头添加中文注释。
5. 复杂业务逻辑、非直觉的条件判断、临时方案（workaround）必须行内注释说明背景和原因。
6. 禁止无意义注释（如 `// 创建用户` 后面跟 `CreateUser()`），注释必须提供代码本身未表达的信息。
7. 接口类型注释必须说明实现方的职责约束和预期行为契约。

### SHOULD
1. TODO/FIXME 注释必须附带责任人和预计回收时间（如 `// TODO(zhangsan): 2026-04 迁移到新接口`）。
2. 注释随代码同步更新；代码逻辑变更后，对应注释必须同步修改，禁止过期注释残留。
3. 包级别注释（`doc.go` 或包头注释）说明包的整体职责和主要用法。

检查方式：`golint` / `revive` 导出注释检查 + 人工审查
阻断级别：阻断合并

## 调试代码清理

### MUST
1. 禁止将 `fmt.Println`、`fmt.Printf`、`log.Println` 等调试打印提交到主分支；所有日志输出必须通过项目统一的结构化日志组件（参见 `common/observability.md`）。
2. 禁止将 `debug`、`spew.Dump`、`pp.Println` 等调试工具调用提交到主分支。
3. 生产构建中禁止包含 `//go:debug` 指令（除非有明确的性能诊断需求并经评审批准）。
4. CI 阶段通过静态分析工具检测并阻断调试代码残留：
   - 推荐使用 `golangci-lint` 配合 `forbidigo` linter，禁止 `fmt.Print*`、`log.Print*`、`spew.*`、`pp.*` 等函数。
   - 配置示例：
   ```yaml
   # .golangci.yml
   linters:
     enable:
       - forbidigo
   linters-settings:
     forbidigo:
       forbid:
         - p: ^fmt\.Print
           msg: “禁止使用 fmt.Print 系列，请使用结构化日志组件”
         - p: ^log\.(Print|Fatal|Panic)
           msg: “禁止使用标准库 log，请使用结构化日志组件”
         - p: ^spew\.
           msg: “禁止提交 spew 调试代码”
   ```
5. 开发环境允许临时使用调试打印，但提交前必须清理；`git diff` 阶段应自查。

检查方式：`golangci-lint`（`forbidigo`）+ CI 阻断
阻断级别：阻断合并

## 分层编码要求
1. 分层依赖必须单向：`transport -> service -> repository`，禁止反向依赖和循环依赖。
2. `transport` 只负责协议适配：请求解析、参数校验、调用 `service`、响应映射。
3. `service` 负责用例编排、事务边界、幂等策略、领域规则与权限策略。
4. `repository` 只负责数据访问与持久化映射，不承载业务决策、鉴权策略或流程编排。
5. 启动层（`main/bootstrap`）只做组装与生命周期管理，禁止承载业务逻辑。
6. 禁止在 handler、main、repository 之间跨层写业务捷径代码。

## 分层边界细则
1. `transport` 禁止直接访问数据库、缓存、对象存储、消息中间件客户端。
2. `transport` 禁止直接引用 `repository`，必须经由 `service` 调用。
3. `service` 禁止依赖 HTTP/gRPC 框架类型和协议层 DTO。
4. `service` 不得直接编写 SQL 或 ORM 细节代码，数据访问必须通过 `repository`。
5. `repository` 禁止处理业务状态机、业务分支决策、跨聚合用例编排。
6. `repository` 可以返回持久化结果，但对外暴露应保持稳定契约，不得泄露驱动层细节。
7. `domain` 模型与规则禁止依赖 `transport`、`repository`、`platform` 的具体实现。

## 模型与 DTO 约束
1. 协议层 DTO 仅用于 `transport`，禁止下沉到 `service`、`repository`。
2. 持久化模型（`repository/model`）用于常规查询与写入，作为数据库结构映射入口。
3. 临时读模型（`repository/query`）仅用于统计分析、多表聚合查询，禁止用于常规 CRUD 写入。
4. 领域对象用于表达业务语义，禁止直接透传持久化模型到外部 API。
5. 不同层模型转换必须显式实现，禁止在单个结构体上混用多层语义标签。

## 事务、错误与日志边界
1. 事务边界定义在 `service` 层，`repository` 仅执行事务上下文内数据操作。
2. 错误处理遵循“内部系统错误记录、对外业务错误映射”原则。
3. `transport` 负责将错误统一映射为稳定响应结构，禁止散落式手写映射逻辑。
4. `repository` 返回错误时必须携带必要上下文并保留根因（`%w`）。
5. 日志记录应在边界层和关键失败点进行，避免同一错误在多层重复刷屏。

## 代码可维护性
1. 每个函数只做一件事，避免超长函数和超深嵌套分支。
2. 公共代码先在模块内部复用，稳定后再考虑提升到共享目录。
3. 并发和资源生命周期必须在代码中可读可验证（超时、取消、关闭顺序）。
4. 对外行为变化必须有测试覆盖：至少覆盖成功路径、参数错误、下游失败、超时或取消场景。

## 分层测试建议
1. `service` 层优先单元测试，使用 mock/fake 验证用例编排和领域规则。
2. `repository` 层优先集成测试，验证 SQL/ORM 行为、事务一致性与索引假设。
3. `transport` 层使用契约测试或接口测试，验证协议、状态码、错误映射和响应结构。
