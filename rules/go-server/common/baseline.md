# rules/go-server/common/baseline.md

## 技术基线
1. Go 版本以 `go.mod` 为准，升级 Go 版本必须单独提交并验证兼容性。
2. 必须使用 Go Modules，禁止依赖 GOPATH 模式。
3. 第三方依赖统一通过 `go.mod`/`go.sum` 管理，禁止复制外部源码进业务目录。
4. 提交前必须确保 `go mod tidy` 后无额外变更。

## 依赖安全审查（MUST）

1. CI 流水线必须集成 `govulncheck`，检测已知漏洞的依赖，发现高危漏洞（CVSS ≥ 7.0）阻断合并。
2. 新增或升级第三方依赖前，必须确认其许可证兼容项目发布方式（商用项目禁止引入 GPL/AGPL 依赖）。
3. 禁止引入已归档（archived）、超过 12 个月无维护更新的依赖，确需使用须在 PR 中说明风险并附回收计划。
4. `go.sum` 必须纳入版本控制，禁止在 `.gitignore` 中忽略。
5. 依赖更新必须单独提交（与业务代码分离），便于审查和回滚。

### SHOULD
1. 定期（每月）执行 `govulncheck ./...` 全量扫描，输出漏洞报告并限时修复。
2. 使用 `go-licenses` 或等效工具自动检测依赖许可证合规性，纳入 CI 检查。
3. 核心依赖（数据库驱动、HTTP 框架、日志库等）锁定主版本，升级需经评审。

检查方式：`govulncheck` + 许可证扫描 + CI 阻断
阻断级别：阻断合并（高危漏洞）/ 告警记录（中低危）

## 基础工程要求
1. 启动入口仅做依赖组装和生命周期管理，不承载业务逻辑。
2. 业务代码必须按分层组织，禁止横向耦合和循环依赖。
3. 跨层函数第一个参数必须是 `context.Context`。
4. 可复用且无业务语义的通用能力放入 `pkg`（如 `pkg/middleware`）。
5. 带作用域语义的能力（如 `admin`、`user`、`open`）放入 `internal`，并按“作用域 + 职责”做到“一文件一责任”。
6. 错误处理必须采用“统一中间件 + 统一响应结构”模式，禁止边界层散落式实现。
7. 组件初始化必须遵循 `common/component-initialization.md`，采用显式依赖注入与统一生命周期管理。
