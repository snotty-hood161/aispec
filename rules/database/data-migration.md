# 数据迁移与种子数据规范

## Skill 协作
1. 数据库相关任务优先使用 `$database-coding-guide`，自动加载本规则。
2. 本规则是 `rules/database/database.md` 的扩展，database.md 中的条款仍然有效。
3. 跨域业务任务使用 `$task-router` 自动路由。

## 迁移脚本规范（MUST）

### 脚本分类

| 类型 | 命名前缀 | 用途 | 可回滚 |
|------|---------|------|--------|
| DDL 迁移 | `yyyyMMdd_VV_` | 表结构、索引、约束变更 | 提供反向脚本 |
| DML 迁移 | `yyyyMMdd_VV_data_` | 数据修正、批量更新 | 提供反向脚本 |
| 种子数据 | `yyyyMMdd_VV_seed_` | 初始化必要的基础数据 | 不需要 |

### 迁移脚本结构（MUST）
每个迁移脚本必须包含以下结构：

```sql
-- Migration: yyyyMMdd_VV_变更说明
-- Author: {作者}
-- Description: {变更描述}
-- Related: {关联的需求/任务编号}

-- ========== UP ==========
-- 正向迁移 SQL

-- ========== DOWN（反向回滚） ==========
-- 回滚 SQL（DDL 和 DML 迁移必须提供）
```

### 迁移脚本规范
1. 每个脚本必须是幂等的（重复执行不会产生副作用）。
2. DDL 变更必须使用 `IF NOT EXISTS` / `IF EXISTS` 防护。
3. 大表变更必须评估锁表影响，超过 100 万行的表变更需标注预估执行时间。
4. 禁止在迁移脚本中使用 `DROP TABLE`（使用重命名 + 保留）。
5. 禁止在迁移脚本中使用 `TRUNCATE`（使用条件 DELETE）。
6. 索引创建在 MySQL 上使用 `ALGORITHM=INPLACE, LOCK=NONE`（如支持）。

### 迁移执行顺序
1. 迁移脚本按文件名排序执行。
2. 已执行的迁移记录在 `_migration_history` 表中。
3. 禁止修改已执行的迁移脚本（同 `database.md` 的核心规则）。
4. 禁止调整已执行脚本的排序。

## 种子数据规范（MUST）

### 种子数据分类

| 类型 | 说明 | 示例 | 环境 |
|------|------|------|------|
| 系统种子 | 系统运行必需的基础数据 | 角色、权限、菜单、字典 | 所有环境 |
| 业务种子 | 业务运行必需的初始数据 | 默认分类、系统配置项 | 所有环境 |
| 测试种子 | 开发和测试用的模拟数据 | 测试用户、示例订单 | dev / test |
| 演示种子 | 产品演示用的样例数据 | 演示账号、演示数据集 | staging（可选） |

### 种子数据规范
1. 系统种子和业务种子包含在 `schema.sql` 中（初始化时一并执行）。
2. 后续新增的种子数据通过迁移脚本（`seed_` 前缀）补充。
3. 测试种子数据存放在独立目录：`docs/seeds/test/`。
4. 种子数据必须是幂等的（`INSERT ... ON DUPLICATE KEY UPDATE` 或 `UPSERT`）。
5. 种子数据禁止包含真实用户的个人信息。
6. 测试种子中的手机号、邮箱、身份证使用虚拟值。

### 种子数据管理
1. 每个种子脚本必须标注：数据用途、目标环境、依赖的表。
2. 种子数据变更纳入 Code Review 流程。
3. 生产环境禁止直接执行测试/演示种子数据。

## 数据备份与恢复（MUST）

### 备份策略

| 环境 | 备份类型 | 频率 | 保留时间 | 存储位置 |
|------|---------|------|---------|---------|
| prod | 全量 + 增量 | 全量每日、增量每小时 | 全量 30 天、增量 7 天 | 异地存储 |
| staging | 全量 | 每周 | 7 天 | 同区域 |
| dev / test | 不强制 | — | — | — |

### 恢复演练（SHOULD）
1. 每季度执行一次数据恢复演练。
2. 记录 RTO（恢复时间目标）和 RPO（恢复点目标）。
3. 恢复演练在 staging 环境执行，禁止在 prod 上演练。

## 数据脱敏（MUST）

### 脱敏场景
1. 生产数据导入 staging / dev / test 环境时必须脱敏。
2. 日志中包含敏感字段时必须脱敏。
3. 数据导出 / 报表中的敏感字段必须脱敏。

### 脱敏规则

| 字段类型 | 脱敏方式 | 示例 |
|---------|---------|------|
| 手机号 | 中间 4 位替换为 * | `138****1234` |
| 身份证 | 中间 10 位替换为 * | `110***********1234` |
| 银行卡 | 仅保留后 4 位 | `**** **** **** 1234` |
| 邮箱 | 用户名部分替换 | `z***g@example.com` |
| 姓名 | 姓保留，名替换 | `张**` |
| 地址 | 省市保留，详细地址替换 | `北京市朝阳区****` |

## 数据库变更发布流程（MUST）

```
变更发布流程：

1. 编写迁移脚本（含 UP 和 DOWN）
2. 本地 dev 环境验证
3. Code Review（至少 1 人审查）
4. test 环境自动执行
5. staging 环境执行并验证
6. 大表变更评估（> 100 万行需审批）
7. prod 发布窗口执行
8. 执行后验证（数据完整性 + 应用兼容性）
9. 记录变更到 _migration_history
```
