# 错误码体系规范

## Skill 协作
1. 前后端协作任务优先使用 `$frontend-backend-coding-guide`，自动加载本规则。
2. 本规则是 `rules/frontend-backend-collaboration.md` 的扩展。
3. 跨域业务任务使用 `$task-router` 自动路由。

## 错误码结构（MUST）

### 编码格式
```
{DOMAIN}_{MODULE}_{CODE}

格式：3 位域前缀 + 下划线 + 2-4 位模块标识 + 下划线 + 3 位数字编号

示例：
SRV_AUTH_001     → 服务端-认证模块-001
SRV_ORDER_002    → 服务端-订单模块-002
CLI_FORM_001     → 客户端-表单模块-001
DB_MIG_001       → 数据库-迁移模块-001
```

### 域前缀

| 前缀 | 域 | 说明 |
|------|-----|------|
| SRV | 服务端 | Go / .NET / Python / Java / Node.js 服务端错误 |
| CLI | 客户端 | 前端 / 移动端 / 桌面端错误 |
| DB | 数据库 | Schema / 迁移 / 约束错误 |
| EXT | 外部服务 | 第三方 API / 支付 / 短信 |
| SYS | 系统 | 基础设施 / 运维错误 |

### 编号范围

| 范围 | 含义 |
|------|------|
| 001-099 | 通用错误（认证、授权、参数校验） |
| 100-199 | 业务逻辑错误 |
| 200-299 | 数据错误（不存在、已存在、数据冲突） |
| 300-399 | 外部依赖错误（第三方 API、支付、短信） |
| 400-499 | 系统错误（超时、限流、降级） |
| 900-999 | 预留（内部调试用） |

## 错误响应格式（MUST）

### API 错误响应

```json
{
  "code": "SRV_ORDER_101",
  "message": "订单金额不能为零",
  "details": {
    "field": "amount",
    "constraint": "must be greater than 0"
  },
  "trace_id": "abc123def456"
}
```

### 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| code | string | 是 | 错误码，客户端通过此字段匹配本地化错误消息 |
| message | string | 是 | 人类可读的默认错误消息（服务端默认语言） |
| details | object | 否 | 错误详情（校验失败的字段、约束条件等） |
| trace_id | string | 是 | 链路追踪 ID，用于排查问题 |

### HTTP 状态码映射

| HTTP 状态码 | 对应场景 | 错误码前缀示例 |
|------------|---------|-------------|
| 400 | 参数校验失败 | `SRV_{MOD}_001~099` |
| 401 | 未认证 | `SRV_AUTH_001` |
| 403 | 无权限 | `SRV_AUTH_002` |
| 404 | 资源不存在 | `SRV_{MOD}_200~299` |
| 409 | 数据冲突 | `SRV_{MOD}_200~299` |
| 422 | 业务规则拒绝 | `SRV_{MOD}_100~199` |
| 429 | 限流 | `SYS_RATE_400` |
| 500 | 服务端内部错误 | `SYS_{MOD}_400~499` |
| 502/503 | 上游服务异常 | `EXT_{MOD}_300~399` |

## 客户端错误码消费（MUST）

### 错误码到文案映射
1. 客户端维护错误码到本地化文案的映射表。
2. 映射表按语言组织，放在国际化文案目录中。
3. 未识别的错误码显示通用错误消息："操作失败，请稍后重试 (错误码: {code})"。

```json
{
  "SRV_AUTH_001": "登录已过期，请重新登录",
  "SRV_AUTH_002": "您没有权限执行此操作",
  "SRV_ORDER_101": "订单金额不能为零",
  "SRV_ORDER_201": "订单不存在"
}
```

### 客户端处理策略

| 错误码前缀 | 客户端行为 |
|-----------|-----------|
| `SRV_AUTH_001` | 跳转登录页 |
| `SRV_AUTH_002` | 显示无权限提示 |
| `SRV_{MOD}_001~099` | 表单字段级错误提示 |
| `SRV_{MOD}_100~199` | Toast 提示 |
| `SYS_RATE_*` | 显示限流提示 + 倒计时 |
| `SYS_*` / `EXT_*` | 显示通用错误提示 + 重试按钮 |

## 错误码管理（MUST）

### 错误码注册流程
1. 新增错误码必须在错误码注册表中登记。
2. 注册表格式：

```markdown
| 错误码 | 模块 | 描述 | HTTP 状态码 | 客户端行为 | 添加版本 |
|--------|------|------|-----------|-----------|---------|
| SRV_AUTH_001 | 认证 | Token 过期或无效 | 401 | 跳转登录 | v1.0.0 |
```

3. 禁止复用已废弃的错误码编号。
4. 错误码变更（含义或行为变更）视同 API 不兼容变更，需要 API 版本升级。

### 错误码审查
1. Code Review 时检查新增错误码的唯一性。
2. 检查错误码与 HTTP 状态码的映射是否合理。
3. 检查客户端是否已添加对应的本地化文案。
