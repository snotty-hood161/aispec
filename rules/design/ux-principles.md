# rules/design/ux-principles.md — 交互设计原则

## 目的
定义 AI 在进行交互设计时必须遵循的用户体验原则，确保产出的交互方案合理、易用、一致。

## 核心原则

### 1. 一致性（Consistency）
- 相同的操作在所有页面中保持相同的行为和外观。
- 术语统一：同一个概念在系统中只用一个词表述。
- 位置固定：全局导航、操作按钮在各页面保持固定位置。
- 反馈统一：成功/失败/加载的反馈方式全局一致。

### 2. 可见性（Visibility）
- 当前状态可见：用户始终知道自己在哪里、在做什么。
- 可用操作可见：用户能看到当前可以执行哪些操作。
- 反馈可见：每个操作都有即时的视觉反馈。
- 进度可见：耗时操作显示进度指示。

### 3. 容错性（Forgiveness）
- 提供撤销机制：危险操作支持撤销或有二次确认。
- 渐进式确认：删除等不可逆操作要求明确确认（如输入确认文字）。
- 错误可恢复：表单提交失败后保留已填内容。
- 自动保存：长表单或编辑操作支持自动保存/草稿。

### 4. 简约性（Simplicity）
- 减少认知负荷：一个页面一个主要任务。
- 渐进式披露：非核心信息折叠或按需展示。
- 合理默认值：为表单提供智能默认值，减少用户输入。
- 最短路径：核心操作在 3 步以内完成。

### 5. 及时反馈（Feedback）
- 即时响应：操作后 100ms 内有视觉反馈。
- 状态明确：成功（绿色）、警告（黄色）、错误（红色）、信息（蓝色）。
- 加载透明：加载时间 > 300ms 显示加载指示器。
- 结果确认：重要操作完成后有明确的结果提示。

## 信息架构原则

### 导航设计
- 一级导航不超过 7±2 项。
- 导航标签使用 2-4 字的简短名词。
- 当前位置在导航中必须有视觉高亮。
- 提供面包屑（层级 > 2 时）。
- 移动端采用底部标签栏（核心入口）+ 汉堡菜单（次要入口）。

### 页面层级
- 最深层级不超过 4 级（首页 → 列表 → 详情 → 编辑）。
- 每级跳转有明确的返回路径。
- 重要功能的入口层级不超过 2（首页即可触达）。

### 内容组织
- 使用卡片或区域分组相关信息。
- 视觉层级与信息重要性一致（大标题 > 小标题 > 正文）。
- 操作按钮靠近操作对象。
- 关联信息在空间上相邻。

## 交互模式规范

### 表单设计
- 标签位置：短表单用顶部标签，长表单用左侧标签。
- 必填标记：使用 `*` 标记必填项，非必填项标注"（选填）"。
- 实时校验：输入框失焦时立即校验，错误信息显示在输入框下方。
- 提交保护：提交按钮在提交中禁用，防止重复提交。
- 错误定位：提交失败后自动滚动到第一个错误字段。

### 列表与表格
- 空列表显示引导操作（"暂无数据，点击创建"）。
- 表格支持排序时，排序方向在表头可见。
- 行操作（编辑/删除）在行末尾或悬停时显示。
- 批量选择时显示已选数量和批量操作栏。
- 分页信息显示总数："共 {n} 条"。

### 搜索与筛选
- 搜索框有清除按钮。
- 搜索结果高亮匹配文字。
- 无结果时提供搜索建议或推荐内容。
- 筛选器支持重置所有筛选条件。
- 活跃的筛选条件以标签形式可见。

### 弹窗与浮层
- 弹窗有明确的关闭方式（关闭按钮 + 点击遮罩 + ESC）。
- 弹窗内容不超过屏幕 80%。
- 表单弹窗未保存时提示"是否放弃修改"。
- 避免弹窗嵌套（弹窗上再弹弹窗）。

### 操作反馈
| 操作类型 | 反馈方式 | 时机 |
|---------|---------|------|
| 创建成功 | Toast + 跳转列表 | 操作完成后 |
| 编辑成功 | Toast | 操作完成后 |
| 删除成功 | Toast + 列表刷新 | 确认后 |
| 提交失败 | 内联错误 + Toast | 立即 |
| 网络异常 | 全局提示 + 重试按钮 | 检测到异常 |
| 权限不足 | 403 页面或 Toast | 立即 |

## 移动端特殊规则
- 触控目标最小 44x44px。
- 重要操作放在拇指热区（屏幕下半部分）。
- 长列表使用下拉刷新 + 上拉加载更多。
- 避免悬停效果（触屏无悬停）。
- 输入框聚焦时页面上推，避免键盘遮挡。
