# rules/frontend/common/workflow.md

## 文档目标
1. 统一从需求到交付的执行流程，降低返工风险。
2. 为高风险操作与大改动提供可追溯确认机制。

## 标准流程（MUST）
1. 需求澄清：确认目标、范围、约束、验收标准。
2. 规则映射：明确适用规则文件与优先级。
3. 实施改动：按最小改动原则交付。
4. 验证回归：执行 `lint + typecheck + test`。
5. 结果归档：记录变更点、风险点、验证结果。
检查方式：人工审查  
阻断级别：告警记录

## 规则按需加载策略（MUST）
1. 每次任务先读取 `rules/frontend/index.md` 作为索引入口。
2. 按任务类型只读取命中规则文件，不得一次性通读 `rules/frontend` 全部文件。
3. 若实施过程中出现新约束，再增量加载相关规则文件。
4. 交付说明中需列出“本次实际读取的规则文件清单”。
检查方式：人工审查  
阻断级别：阻断合并

## 页面与组件交付检查（MUST）
1. `index.vue` 文件超过 `200` 行需预警并评估拆分。
2. `index.vue` 与页面私有组件超过 `300` 行必须拆分后再合并。
3. 页面容器层只做编排，复杂区块拆到私有组件。
4. 涉及样式改动时必须说明是否复用 token。
检查方式：人工审查 + 静态扫描  
阻断级别：阻断合并

## tab-host 场景流程（适用时 MUST）
1. 当页面为多 Tab 宿主页模式时，优先使用 `tab-host` 容器管理切换。
2. Tab 子视图优先用组件切换，避免无必要的路由跳转。
3. 需深链路由时，必须说明为何不使用 `tab-host` 模式。
检查方式：人工审查  
阻断级别：告警记录

## 危险操作确认机制（MUST）
1. 删除文件/目录或大规模重构前必须确认。
2. 不可逆操作（强制 push、硬重置等）前必须确认。Git 分支与提交规范详见 `common/git-workflow.md`。
3. 生产配置与权限变更前必须确认。
检查方式：人工审查  
阻断级别：阻断合并

确认模板：
```text
【危险操作检测】
操作类型：
影响范围：
风险等级：
潜在后果：

请确认是否继续？（是/确认/继续）
```
