# rules/frontend/common/governance.md

## 文档目标
1. 定义前端规范的规则分级、治理流程、版本策略与执行机制。
2. 确保规则可维护、可审计、可执行、可追溯。
3. 适用于 `rules/frontend/` 下全部规则文件与演进流程。
4. 例外申请使用统一模板，参见 `rules/templates/exception-request-template.md`。

---

## 一、规则分级定义

### 1.1 级别总览

| 级别 | 含义 | 违反后果 | CI 行为 | 合并影响 |
|------|------|----------|---------|----------|
| **MUST** | 强制执行，无条件遵守 | 阻断合并，必须修复 | 报错（Error） | 禁止合并 |
| **SHOULD** | 推荐执行，有条件豁免 | 评审警告，需记录原因 | 告警（Warning） | 允许合并，但需登记技术债 |
| **MAY** | 可选实践，视场景采用 | 不阻断，建议采纳 | 不检查 | 不影响 |

### 1.2 MUST（强制）

1. 违反 MUST 条款的代码 **禁止合并到主分支**。
2. 每条 MUST 规则必须满足以下条件：
   - 标注 **检查方式**（人工审查 / 静态扫描 / CI 阻断，至少一个）。
   - 标注 **阻断级别**（阻断合并）。
   - 能被自动化检查的必须配置到 CI 流水线。
3. MUST 规则的新增、修改、降级必须经 **技术负责人审批**。
4. 任何人不得在 PR 评审中自行豁免 MUST 条款，豁免必须走例外申请流程。

### 1.3 SHOULD（推荐）

1. 违反 SHOULD 条款允许合并，但必须满足以下条件：
   - PR 描述中 **记录违反原因**。
   - 创建技术债 Issue，标注 **回收时间**（不超过 2 个迭代周期）。
   - 指定 **回收责任人**。
2. 每条 SHOULD 规则至少标注人工审查方式。
3. 连续 3 次季度复盘中无人违反的 SHOULD 规则，可考虑升级为 MUST。
4. 连续 3 次季度复盘中频繁被豁免的 SHOULD 规则，应评估是否降级为 MAY 或修改规则内容。

### 1.4 MAY（可选）

1. MAY 条款为最佳实践建议，不纳入 CI 检查，不影响合并。
2. 评审中鼓励提出 MAY 建议，但不作为阻塞理由。
3. 当某 MAY 实践在团队中广泛采用（> 80% 的项目自发遵守）时，应评估升级为 SHOULD。

## 二、规则 ID 命名规范

### 2.1 格式

```
{域}-{模块}-{序号}
```

| 字段 | 含义 | 取值 |
|------|------|------|
| 域 | 规范域 | `FE`（前端）、`GO`（Go 服务端）、`COLLAB`（前后端协作） |
| 模块 | 技术/业务模块缩写 | 见下方字典 |
| 序号 | 三位数字，从 001 递增 | `001` ~ `999` |

### 2.2 模块缩写字典（前端）

| 缩写 | 模块 | 示例 |
|------|------|------|
| `TS` | TypeScript 基线 | `FE-TS-001` |
| `VUE` | Vue3 框架规则 | `FE-VUE-010` |
| `RCT` | React 框架规则 | `FE-RCT-005` |
| `CMP` | 组件化与适配 | `FE-CMP-003` |
| `STY` | 样式与 Token | `FE-STY-002` |
| `API` | 接口与状态 | `FE-API-001` |
| `PERF` | 性能 | `FE-PERF-004` |
| `SEC` | 安全 | `FE-SEC-001` |
| `TOOL` | 工具链与 CI | `FE-TOOL-002` |
| `STRUC` | 项目结构 | `FE-STRUC-001` |
| `NAME` | 命名规范 | `FE-NAME-003` |
| `FORM` | 表单（Schema-Driven） | `FE-FORM-001` |
| `PERM` | 权限 | `FE-PERM-002` |
| `MON` | 监控与埋点 | `FE-MON-001` |
| `WX` | 微信生态 | `FE-WX-005` |
| `MP` | 小程序平台 | `FE-MP-001` |

### 2.3 规则

1. **MUST**：新增规则必须分配唯一规则 ID，禁止重复。
2. **MUST**：规则废弃后 ID 不得复用，标记为 `DEPRECATED`。
3. **SHOULD**：规则文件中每条 MUST/SHOULD 条款建议附带规则 ID，便于引用和追溯。

## 三、规则模板（统一格式）

每条规则的完整描述应包含以下字段：

```markdown
### {规则 ID}：{规则标题}

- **级别**：MUST / SHOULD / MAY
- **适用范围**：{框架/应用端/目录/版本}
- **规则内容**：
  - 正向要求：{做什么}
  - 反向禁止：{不做什么}
- **检查方式**：人工审查 / 静态扫描 / CI 阻断
- **阻断级别**：阻断合并 / 告警记录 / 观察项
- **示例**：{正确写法 vs 错误写法}
```

**说明**：实际规则文件中不要求每条都写完整模板，但 MUST 级别条款至少包含「检查方式」和「阻断级别」。

## 四、检查方式字典

| 检查方式 | 说明 | 执行时机 | 示例 |
|----------|------|----------|------|
| **人工审查** | 代码评审时由 Reviewer 逐项核对 | PR 提交后 | PR Review Checklist |
| **静态扫描** | 自动化工具扫描代码或配置 | 本地 + CI | ESLint、TypeScript 编译、依赖扫描 |
| **CI 阻断** | CI 流水线检查不通过时禁止合并 | PR 合并前 | `lint`、`typecheck`、`test`、体积检查 |
| **架构评审** | 涉及架构决策的重大变更由架构师评审 | 设计阶段 | 新增公共库、技术选型变更 |

## 五、例外申请流程

### 5.1 适用场景

1. 技术限制导致暂时无法遵守某条 MUST 规则。
2. 第三方依赖约束导致必须偏离规范。
3. 紧急修复需要临时绕过某条规则。

### 5.2 流程

```
申请人提交例外申请
       │
       ▼
技术负责人评审
       │
       ├── 驳回 → 申请人按规范修改
       │
       └── 批准 → 记录例外信息
                    │
                    ▼
              设定回收计划
              （最长 3 个月）
                    │
                    ▼
              定期复查（每月）
                    │
                    ├── 已回收 → 关闭例外
                    └── 未回收 → 升级提醒
                                  │
                                  └── 超期未回收 → 技术负责人介入
```

### 5.3 规则

1. **MUST**：例外申请使用统一模板（`rules/templates/exception-request-template.md`）。
2. **MUST**：例外有效期最长 3 个月，超期必须续期或回收。
3. **MUST**：每个例外必须指定回收责任人和验证标准。
4. **MUST**：跨端影响的例外需架构师额外审批。
5. **SHOULD**：紧急修复的例外在事后 3 个工作日内补交申请。

## 六、治理流程

### 6.1 规范生命周期

```
提出 → 评审 → 试行 → 正式发布 → 执行 → 复盘 → 迭代/废弃
```

### 6.2 各阶段要求

| 阶段 | 负责人 | 产出 | 要求 |
|------|--------|------|------|
| **提出** | 任何团队成员 | Issue / RFC 文档 | 说明问题背景、收益、影响面 |
| **评审** | 技术负责人 + 相关方 | 评审记录 | MUST 规则需至少 2 人同意 |
| **试行** | 规则提出人 | 试行报告 | 在 1 个项目中试行 1-2 个迭代 |
| **正式发布** | 技术负责人 | 版本变更记录 | 更新规则文件、版本号、索引 |
| **执行** | 全体开发者 | CI 检查 + PR 审查 | 配置自动化检查（能自动化的） |
| **复盘** | 技术负责人 | 季度复盘报告 | 命中率、误报率、维护成本 |
| **迭代/废弃** | 技术负责人 | 变更 PR | 升级/降级/废弃需评审 |

### 6.3 变更规则

1. **MUST**：MUST 规则的新增、修改、降级、废弃必须单独提交 PR 并附变更说明。
2. **MUST**：破坏性变更（MUST 规则变动）必须提前通知所有相关项目，给予迁移窗口。
3. **SHOULD**：SHOULD / MAY 规则变更可合并在其他 PR 中，但需在变更记录中体现。

## 七、版本策略

### 7.1 版本号格式

```
v{主版本}.{次版本}.{修订版本}
```

| 类型 | 触发条件 | 示例 |
|------|----------|------|
| **主版本** | MUST 规则有破坏性变更（新增/删除/修改约束） | `v1.0.0` → `v2.0.0` |
| **次版本** | 新增 SHOULD/MAY 规则、新增模板、非破坏性调整 | `v1.0.0` → `v1.1.0` |
| **修订版本** | 错别字、示例修正、说明澄清 | `v1.1.0` → `v1.1.1` |

### 7.2 规则

1. **MUST**：每次发布必须更新 `CHANGELOG.md` 变更记录。
2. **MUST**：主版本升级必须附迁移指南，说明现有项目如何适配。
3. **SHOULD**：变更记录格式遵循 [Keep a Changelog](https://keepachangelog.com/) 风格。

## 八、季度复盘机制

### 8.1 复盘内容

| 指标 | 说明 | 健康标准 |
|------|------|----------|
| **MUST 违反率** | 季度内 MUST 规则在 PR 中被违反的次数 | ≤ 5% |
| **SHOULD 豁免率** | 季度内 SHOULD 规则被豁免的比例 | ≤ 20% |
| **例外回收率** | 已过期例外中按时回收的比例 | ≥ 90% |
| **CI 误报率** | 自动化检查产生误报的比例 | ≤ 3% |
| **规则覆盖率** | 已配置自动化检查的 MUST 规则占比 | ≥ 80% |

### 8.2 复盘行动

1. MUST 违反率超标 → 排查是否规则不合理或工具不到位，调整规则或增强检查。
2. SHOULD 豁免率超标 → 评估规则是否脱离实际，考虑降级为 MAY 或修改内容。
3. 例外回收率不达标 → 技术负责人逐个跟进未回收项，必要时升级处理。
4. CI 误报率超标 → 优化检查规则或工具配置，降低误报对开发效率的影响。
