# rules/python-server/profiles/microservice/gateway.md

## 文档目标
1. 定义 API 网关的职责边界与约束。仅在架构包含网关层时启用本文件。

---

## 网关职责（MUST）

1. 网关必须且仅承担以下职责：
   - 路由转发（请求路由到目标服务）
   - 协议转换（HTTP ↔ gRPC，若需要）
   - 鉴权（统一认证，Token 校验）
   - 限流（全局/接口级限流）
   - 日志与追踪（注入 trace ID，记录访问日志）
2. 网关禁止包含业务逻辑（数据聚合、字段转换、业务校验）。
3. 网关路由配置必须版本化，与服务契约保持同步。

检查方式：架构评审
阻断级别：阻断合并

---

## 网关选型与部署（MUST）

1. 推荐方案：`Kong`、`APISIX`、`Envoy`、`Traefik`，或自建轻量网关。
2. Python 自建网关推荐使用 FastAPI 作为 BFF（Backend for Frontend）层，但 BFF 仅做聚合和协议适配，禁止包含核心业务逻辑。
3. gRPC ↔ HTTP 转换推荐使用 `grpc-gateway` 或 `Envoy` 代理，禁止手写转换逻辑。
4. 网关必须高可用部署（至少 2 实例），并配置健康检查。

### Python BFF 层约束
1. BFF 层可聚合多个下游服务的响应，但聚合逻辑仅限于组装和格式转换。
2. BFF 层禁止直接访问数据库，数据必须通过下游服务获取。
3. BFF 层的下游调用必须配置超时和熔断（参见 `resilience.md`）。
4. BFF 层必须透传 trace 上下文和 request_id。

### SHOULD
1. 网关配置通过配置中心管理，支持动态更新路由规则。
2. 网关指标（QPS、延迟、错误率）纳入统一监控。

检查方式：架构评审
阻断级别：阻断合并

---

## 网关安全（MUST）

1. 网关必须实施认证前置：未经认证的请求禁止路由到后端服务（白名单路由除外，如健康检查、公开 API）。
2. 网关必须实施速率限制，防止单一客户端或 IP 过度消耗资源。
3. 网关必须记录完整的访问日志：请求时间、来源 IP、请求方法/路径、响应状态码、耗时。
4. 网关到后端服务的通信推荐使用内网 TLS 或 mTLS。

检查方式：安全审查
阻断级别：阻断合并
