# 数据库 PR 评审清单模板

## 文档目标
1. 用于涉及数据库 Schema 或迁移脚本变更的 PR 评审，评审人逐项核对，确保变更安全可控。
2. 结合 `database.md` 和 `data-migration.md` 中的规则要求。

## 使用方式
1. **谁用**：PR 评审人（Reviewer）。
2. **何时用**：PR 包含 `schema.sql` 变更、新增迁移脚本、种子数据变更时。
3. **怎么用**：复制清单到 PR 评审评论中，逐项勾选，未通过项写明阻塞原因。

## 前提
1. 评审前先标注变更类型：`DDL`（结构变更）/ `DML`（数据变更）/ `seed`（种子数据）。
2. 每项必须给出结果：`[x]` 通过 / `[ ]` 不通过（需写阻塞原因）。
3. 如有例外，必须在 PR 说明中记录：原因、边界、回收时间。

## 优先级说明
1. `P0` 为阻塞项，必须全部通过才可合并。
2. `P1` 为改进项，允许带条件合并，但必须登记技术债与回收计划。
3. 评审结论遵循：任一 `P0` 未通过则 `Request Changes`。

---

## PR 基本信息
- [ ] [P0] 已标注变更类型：`DDL` / `DML` / `seed`
- [ ] [P0] 已说明变更目的、影响范围、回滚方案
- [ ] [P0] 已附关联的需求/任务编号

## 脚本不可变性
- [ ] [P0] 未修改 `schema.sql` 中的已有内容
- [ ] [P0] 未修改任何已存在的历史迁移脚本
- [ ] [P0] 所有变更通过新增迁移脚本完成

## 迁移脚本规范
- [ ] [P0] 文件命名符合格式：`yyyyMMdd_VV_变更说明.sql`
- [ ] [P0] 脚本包含完整头部注释（Migration / Author / Description / Related）
- [ ] [P0] DDL 和 DML 迁移包含 `UP` 和 `DOWN` 两段
- [ ] [P0] 脚本是幂等的（重复执行不会产生副作用）
- [ ] [P0] DDL 变更使用 `IF NOT EXISTS` / `IF EXISTS` 防护

## 安全与风险
- [ ] [P0] 未使用 `DROP TABLE`（应使用重命名 + 保留）
- [ ] [P0] 未使用 `TRUNCATE`（应使用条件 DELETE）
- [ ] [P0] 无 `SELECT *`，查询使用参数化
- [ ] [P0] 大表变更（> 100 万行）已标注预估执行时间
- [ ] [P1] 大表索引创建使用 `ALGORITHM=INPLACE, LOCK=NONE`（如适用）

## 种子数据（如涉及）
- [ ] [P0] 种子数据是幂等的（`ON DUPLICATE KEY UPDATE` 或 `UPSERT`）
- [ ] [P0] 种子数据未包含真实用户个人信息
- [ ] [P0] 测试/演示种子数据存放在 `docs/seeds/test/`，未混入生产种子
- [ ] [P1] 种子脚本标注了数据用途、目标环境、依赖的表

## 数据脱敏（如涉及数据导出或跨环境）
- [ ] [P0] 生产数据导入非生产环境时已脱敏
- [ ] [P0] 日志中的敏感字段已脱敏
- [ ] [P1] 脱敏规则符合规范（手机号/身份证/银行卡/邮箱/姓名/地址）

## 发布流程
- [ ] [P0] 已在本地 dev 环境验证通过
- [ ] [P1] 已在 test/staging 环境验证通过
- [ ] [P0] 迁移执行顺序正确（按文件名排序，未调整已执行脚本的排序）

---

## 结论
- [ ] `Approve`（全部 `P0` 通过）
- [ ] `Request Changes`（存在任一 `P0` 未通过）
- [ ] `Conditional Approve`（`P0` 通过，存在 `P1` 未通过且已登记技术债）
