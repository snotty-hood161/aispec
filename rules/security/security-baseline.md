# 安全基线规范

## Skill 协作
1. 安全审计任务优先使用 `$security-auditor`，自动加载本规则。
2. 各域 coding-guide 中的 security.md 为域内安全细则，本文件为跨域安全基线。
3. 跨域业务任务使用 `$task-router` 自动路由。

## 认证与授权（MUST）

### 认证
1. 密码存储必须使用 bcrypt / scrypt / Argon2，禁止 MD5 / SHA-1。
2. JWT 签名必须使用 RS256 或 ES256，禁止 HS256 在分布式场景使用。
3. Token 有效期：Access Token ≤ 30 分钟，Refresh Token ≤ 7 天。
4. 失败登录锁定：连续失败 5 次后锁定 15 分钟，支持 CAPTCHA 二次验证。
5. 敏感操作必须二次验证（密码修改、支付、权限变更）。

### 授权
1. 默认拒绝：未明确授权的资源一律返回 403。
2. API 鉴权：所有非公开 API 必须携带有效 Token。
3. 资源隔离：用户只能访问自己的数据，多租户必须强制租户隔离。
4. 权限最小化：服务账号和数据库连接只授予必要权限。

## 数据安全（MUST）

### 传输安全
1. 所有外部通信必须使用 TLS 1.2+，推荐 TLS 1.3。
2. 禁止 HTTP 明文传输敏感数据。
3. 内部微服务间通信推荐 mTLS。

### 存储安全
1. 敏感字段（身份证、手机号、银行卡）必须加密存储或脱敏。
2. 加密密钥禁止与数据存储在同一位置。
3. 数据库备份必须加密。
4. 日志中禁止打印敏感信息（密码、Token、身份证号）。

### 数据分类
| 分类 | 示例 | 存储要求 | 传输要求 | 日志要求 |
|------|------|---------|---------|---------|
| 绝密 | 密码、密钥、证书 | AES-256 加密 | TLS + 额外加密层 | 禁止记录 |
| 机密 | 身份证、银行卡、手机号 | AES-256 或脱敏 | TLS | 脱敏后可记录 |
| 内部 | 用户名、邮箱、订单详情 | 明文可存 | TLS | 可记录 |
| 公开 | 产品名、公告 | 明文可存 | 可 HTTP | 可记录 |

## 输入校验（MUST）
1. 所有用户输入必须在服务端校验，不信任客户端校验结果。
2. SQL 查询必须使用参数化查询 / ORM，禁止字符串拼接。
3. HTML 输出必须转义，防止 XSS。
4. 文件上传必须校验类型、大小、内容（不仅校验扩展名）。
5. URL 重定向必须使用白名单校验，防止开放重定向。
6. JSON / XML 解析必须限制深度和大小，防止 DoS。

## API 安全（MUST）
1. 接口限流：公开 API ≤ 100 次/分钟/IP，认证 API ≤ 300 次/分钟/用户。
2. CORS 配置：生产环境禁止 `Access-Control-Allow-Origin: *`。
3. 安全响应头必须配置：
   - `X-Content-Type-Options: nosniff`
   - `X-Frame-Options: DENY`（或 `SAMEORIGIN`）
   - `Strict-Transport-Security: max-age=31536000; includeSubDomains`
   - `Content-Security-Policy`（按应用需求配置）
4. 错误响应禁止暴露内部实现细节（堆栈、SQL 语句、文件路径）。

## 密钥管理（MUST）
1. 禁止在代码库中硬编码密钥、密码、Token、证书。
2. 密钥必须通过环境变量或密钥管理服务（Vault / KMS）注入。
3. .env 文件必须在 .gitignore 中排除。
4. CI/CD 中的密钥使用 Secrets 管理（GitHub Secrets / GitLab CI Variables）。
5. 密钥必须定期轮换，轮换周期 ≤ 90 天。
6. 旧密钥轮换后必须在 24 小时内失效。

## 安全审计日志（MUST）
1. 必须记录的事件：登录成功/失败、权限变更、敏感数据访问、管理员操作。
2. 日志格式包含：时间戳、用户 ID、操作类型、目标资源、来源 IP、结果。
3. 审计日志禁止修改和删除，保留期 ≥ 180 天。
4. 异常行为告警：短时间大量失败登录、异常 IP 访问、权限提升操作。

## 依赖安全（MUST）
1. 生产依赖禁止使用已知存在高危漏洞（CVSS ≥ 7.0）的版本。
2. 依赖版本必须锁定（lockfile 提交到版本库）。
3. CI 流水线必须包含依赖漏洞扫描步骤。
4. 高危漏洞修复 SLA：严重（CVSS ≥ 9.0）24 小时，高危（CVSS ≥ 7.0）48 小时。

## 安全事件响应（SHOULD）
1. 制定安全事件分级标准（P0 ~ P3）。
2. 各级别对应的响应时间和升级路径。
3. 事件响应后必须产出复盘报告，更新安全基线。
4. 定期进行安全演练（建议每季度一次）。
