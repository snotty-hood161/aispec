# API 版本管理规范

## Skill 协作
1. 前后端协作任务优先使用 `$frontend-backend-coding-guide`，自动加载本规则。
2. 本规则是 `rules/frontend-backend-collaboration.md` 的扩展，协作规范中的条款仍然有效。
3. 跨域业务任务使用 `$task-router` 自动路由。

## 版本策略（MUST）

### 版本标识方式
采用 URL 路径版本方案：

```
https://api.example.com/v1/orders
https://api.example.com/v2/orders
```

| 方案 | 格式 | 适用场景 | 本规范推荐 |
|------|------|---------|-----------|
| URL 路径 | `/v1/resource` | 公开 API、对外服务 | 推荐（默认方案） |
| 请求头 | `Accept: application/vnd.api.v1+json` | 内部微服务 | 可选 |
| 查询参数 | `?api-version=1` | 临时方案 | 不推荐 |

### 版本号规则
1. 主版本号从 `v1` 开始，仅在不兼容变更时递增。
2. 次版本变更（向后兼容）不改变 URL 中的版本号，通过 API 响应头 `X-API-Version: 1.2.0` 标识。
3. 内部 API 可不带版本号，但必须在 API 契约文档中标注版本。

## 兼容性规则（MUST）

### 向后兼容的变更（不需要新版本）
- 新增可选的请求参数。
- 新增响应字段。
- 新增 API 端点。
- 新增枚举值（客户端应忽略未知枚举）。
- 放宽参数校验规则（如长度上限增加）。

### 不兼容的变更（需要新版本）
- 移除或重命名现有字段。
- 修改字段类型。
- 修改已有字段的语义。
- 收紧参数校验规则。
- 修改错误码的含义。
- 修改认证/授权机制。

### 兼容性保障
1. 新增字段必须有默认值，不影响现有客户端。
2. 枚举类型扩展时，客户端必须能容忍未知值（忽略或使用 fallback）。
3. 响应中废弃的字段先标记 `deprecated`，保留至少 2 个版本后移除。

## 版本生命周期（MUST）

### 版本状态

| 状态 | 说明 | API 行为 |
|------|------|---------|
| current | 当前推荐版本 | 正常响应 |
| deprecated | 已废弃，仍可用 | 正常响应 + `Sunset` 响应头 |
| retired | 已下线 | 返回 410 Gone + 迁移指引 |

### 废弃流程
1. 废弃公告：通过文档、邮件、API 响应头通知使用方。
2. 废弃响应头：
   ```
   Deprecation: true
   Sunset: Sat, 01 Jan 2027 00:00:00 GMT
   Link: <https://docs.example.com/migration/v1-to-v2>; rel="successor-version"
   ```
3. 过渡期：废弃到下线至少保留 6 个月（公开 API）或 3 个月（内部 API）。
4. 监控废弃 API 调用量，当调用量降至 < 1% 时可申请下线。

### 版本并行
1. 同时活跃的主版本不超过 2 个（current + deprecated）。
2. 新版本上线后，旧版本立即进入 deprecated 状态。
3. 所有版本共享同一数据存储，通过适配层做版本转换。

## 迁移指南规范（MUST）

每次主版本升级必须提供迁移指南：

```markdown
# API v1 → v2 迁移指南

## 变更摘要
- {一段话描述本次升级的核心变更}

## 不兼容变更清单
| 端点 | 变更类型 | v1 行为 | v2 行为 | 迁移方法 |
|------|---------|--------|--------|---------|
| GET /orders | 字段重命名 | `order_no` | `order_number` | 替换字段名 |

## 新增功能
- {列出 v2 新增的端点和功能}

## 时间线
- v2 上线日期：{yyyy-MM-dd}
- v1 废弃日期：{yyyy-MM-dd}
- v1 下线日期：{yyyy-MM-dd}

## 客户端迁移步骤
1. {步骤 1}
2. {步骤 2}
```

## API 文档版本管理（SHOULD）
1. API 文档与代码同步更新，使用 OpenAPI / Swagger 规范。
2. 文档按版本组织，每个版本有独立的文档页面。
3. 文档标注每个端点的版本引入时间和废弃状态。
4. 提供版本间差异对比视图。
