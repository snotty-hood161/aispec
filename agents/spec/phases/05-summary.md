# Phase 5 — 汇总输出（Summary）

## 阶段目标
汇总前四个阶段的所有产出，生成完整的 Spec 文档集，执行一致性检查，输出最终交付物。

## 一致性检查清单

### 5.1 模块接口对齐检查
- 模块 A 调用模块 B 的接口 → 该接口必须在模块 B 的 API Spec 中定义。
- 调用方的请求参数类型/名称必须与被调用方的定义一致。
- 错误码引用必须在被调用方的错误码清单中存在。

### 5.2 数据结构一致性检查
- 跨模块引用的实体（如订单模块引用用户 ID）必须使用相同的类型和约束。
- DTO 中引用的外部实体字段必须在源模块中存在。
- 枚举值定义在引用方和定义方之间保持一致。

### 5.3 技术选型兼容性检查
- Phase 2 的技术决策不与 Phase 3 的约束冲突（如选了 SQLite 但要求高并发写入）。
- 架构模式与部署约束兼容（如微服务架构要求容器化部署能力）。
- 认证方案与多端场景兼容（如 Session 方案在移动端的适配）。

### 5.4 业务规则完备性检查
- 每个状态机的状态转换是否覆盖所有合法路径。
- 每个业务规则是否有对应的边界条件定义。
- 权限模型是否覆盖所有 API 接口。
- 错误码是否覆盖所有已知的错误场景。

### 5.5 里程碑覆盖检查
- 所有 P0 模块是否包含在 MVP 里程碑中。
- 模块间依赖是否影响里程碑的执行顺序。
- 每个里程碑的交付物是否明确。

## 风险评估

对整个 Spec 执行风险评估，输出风险清单：

```
### 风险清单

| 风险编号 | 风险描述 | 影响范围 | 概率 | 影响 | 缓解方案 |
|---------|---------|---------|------|------|---------|
| RISK-001 | 第三方支付 API 不稳定 | 订单模块 | 中 | 高 | 备用支付通道 + 降级方案 |
| RISK-002 | 团队无 Go 经验 | 全部后端 | 高 | 中 | 安排培训 + 引入有经验的技术顾问 |
```

## 技术决策记录（ADR）

将 Phase 2 的所有技术决策整理为 ADR（Architecture Decision Record）格式：

```
### ADR-{编号}：{决策主题}

- 状态：{已采纳 / 待定 / 已弃用}
- 日期：{决策日期}
- 决策者：{参与决策的人}
- 上下文：{为什么需要做这个决策}
- 考虑的方案：{方案 A / B / C 的简要描述}
- 决策：{最终选择的方案}
- 理由：{选择该方案的核心理由}
- 影响：{该决策对系统的影响}
- 风险：{潜在风险和缓解方案}
```

## 最终交付物

### 文档清单

| 文档 | 内容 | 格式 |
|------|------|------|
| 项目级 Spec | 系统目标、技术选型、全局约束 | 按 `templates/project-spec-template.md` |
| 模块级 Spec × N | 每个模块的 12 维度详细定义 | 按 `templates/module-spec-template.md` |
| 模块依赖图 | 模块间依赖关系可视化 | 文本/Mermaid 图 |
| 技术决策记录 | 所有 ADR 的汇总 | ADR 格式 |
| 风险清单 | 已识别风险及缓解方案 | 表格 |
| 里程碑计划 | 阶段性目标和交付物 | 甘特图/表格 |

### 里程碑计划输出格式

```
### 里程碑计划

| 里程碑 | 时间 | 目标 | 包含模块 | 交付物 | 验收标准 |
|--------|------|------|---------|--------|---------|
| M1 — MVP | {时间} | 核心功能可用 | 用户、订单、支付 | 可部署的最小系统 | {验收标准} |
| M2 — 增强 | {时间} | P1 功能补齐 | 通知、报表、运营 | 完整功能系统 | {验收标准} |
```

## Coordinator 交接

Spec Agent 完成后，将以下信息交接给 Coordinator：

```
## Spec 交接摘要
- 系统概述：{一段话描述}
- 技术栈：{服务端 + 客户端 + 数据库 + 中间件}
- 架构模式：{架构选择}
- 模块清单：{模块名列表}
- 建议执行顺序：{基于模块依赖的开发顺序}
- 全局约束摘要：{关键约束条件}
- 风险提示：{P0 风险清单}
```

Coordinator 基于此摘要调度后续域 Agent：
1. Database Agent — 根据各模块的数据结构 Spec 设计 Schema。
2. GoServer / DotnetServer / PythonServer / JavaServer / NodeServer Agent — 根据 API 接口 Spec 实现服务端。
3. Collaboration Agent — 根据 API Spec 校验前后端契约。
4. 客户端 Agent — 根据模块 Spec + API 契约实现前端/移动端。

## 输出约束
1. 所有输出使用中文。
2. 文件路径使用相对于项目根目录的格式。
3. 技术术语首次出现时附中文解释。
4. 表格优先于大段文字描述。
5. 每个 Spec 文档可独立阅读，不过度依赖其他文档的上下文。
