# Phase 2 — 技术决策（Decisions）

## 阶段目标
逐项完成关键技术选型。每个决策点给出 1-3 个备选方案，包含问题分析、推荐理由和优缺点对比，由用户最终拍板。

## 决策原则
1. **上下文驱动** — 所有推荐必须基于 Phase 1 收集的业务信息（团队规模、用户规模、预算等）。
2. **不做空谈** — 每个方案必须回答"解决什么问题"和"代价是什么"。
3. **可逆性优先** — 优先推荐可逆（低切换成本）的方案；不可逆决策需要特别标注。
4. **组合兼容** — 后续决策需考虑前序决策的约束（如选了 Go 后端则排除 .NET 专属的方案）。

## 技术决策清单

### 2.1 架构模式
**要解决的问题**：系统整体以什么形态组织和部署？

需考虑的因素：
- 团队规模和组织结构
- 系统复杂度和模块耦合度
- 部署环境和运维能力
- 预期的扩展方向

典型方案空间：
- 单体架构（Monolith）
- 模块化单体（Modular Monolith）
- 微服务（Microservices）
- Serverless / FaaS
- 混合架构（核心单体 + 边缘微服务）

### 2.2 服务端语言与框架
**要解决的问题**：用什么语言和框架实现后端业务逻辑？

需考虑的因素：
- 团队技术栈储备
- 性能需求（CPU 密集 / IO 密集）
- 生态成熟度（第三方库、工具链）
- 长期维护成本和招聘难度

### 2.3 客户端技术栈
**要解决的问题**：用户通过什么形式使用系统？

需考虑的因素：
- 目标平台（Web / iOS / Android / 桌面 / 小程序）
- 跨平台需求（一套代码 vs 原生体验）
- 离线能力需求
- 用户体验要求层级

### 2.4 数据库选型
**要解决的问题**：系统的核心数据如何持久化存储？

需考虑的因素：
- 数据模型（关系型 / 文档型 / 图 / 时序）
- 读写比例和并发量
- 事务要求和一致性级别
- 运维复杂度和成本

### 2.5 缓存方案
**要解决的问题**：如何减少数据库压力、降低接口延迟？

需考虑的因素：
- 缓存粒度（接口级 / 数据级 / 页面级）
- 缓存一致性策略（Cache-Aside / Write-Through / Write-Behind）
- 分布式缓存 vs 本地缓存
- 缓存穿透/击穿/雪崩防护

### 2.6 消息队列与事件总线
**要解决的问题**：系统内部或系统间如何进行异步通信？

需考虑的因素：
- 消息可靠性要求（At-least-once / Exactly-once）
- 消息顺序性要求
- 吞吐量需求
- 运维复杂度
- 是否需要事件溯源

### 2.7 认证与授权方案
**要解决的问题**：如何验证用户身份、控制访问权限？

需考虑的因素：
- 认证方式（Session / JWT / OAuth2 / OIDC）
- 多端统一登录需求
- 第三方登录集成
- 权限粒度（角色级 / 资源级 / 字段级）
- 多租户需求

### 2.8 API 风格
**要解决的问题**：前后端通过什么协议和风格交互？

需考虑的因素：
- 客户端类型（Web / Mobile / IoT / 第三方开发者）
- 数据查询复杂度
- 强类型需求
- 实时通信需求

典型方案空间：
- RESTful API
- gRPC（内部服务间通信）
- GraphQL
- WebSocket（实时场景）
- 混合（REST + WebSocket / REST + gRPC）

### 2.9 文件存储方案
**要解决的问题**：用户上传的文件、系统生成的文件如何存储和分发？

需考虑的因素：
- 文件类型和大小范围
- 访问频率和分发策略（CDN）
- 安全需求（私有文件 / 签名 URL）
- 存储成本

### 2.10 搜索方案（如需要）
**要解决的问题**：系统是否需要全文搜索或复杂查询能力？

需考虑的因素：
- 搜索场景和复杂度
- 数据量级
- 实时性要求
- 多语言/分词需求

## 决策输出格式

每个决策点使用以下标准格式输出：

```
### 技术决策 #{编号}：{决策主题}
- 决策类型：{可逆 / 不可逆}
- 要解决的问题：{一句话描述}
- 上下文依赖：{引用 Phase 1 的哪些信息影响了本决策}

#### 方案 A：{方案名}
- 解决什么问题：{核心痛点}
- 推荐理由：{为什么在当前上下文下值得考虑}
- 优点：
  - {优点 1}
  - {优点 2}
- 缺点：
  - {缺点 1}
  - {缺点 2}
- 适用场景：{最适合的条件}
- 参考案例：{已知的成功案例}

#### 方案 B：{方案名}
（同上结构）

#### 方案 C：{方案名}（如有）
（同上结构）

#### 方案对比矩阵

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 开发效率 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| 性能 | ... | ... | ... |
| 运维成本 | ... | ... | ... |
| 学习曲线 | ... | ... | ... |
| 社区/生态 | ... | ... | ... |

→ 推荐选择：方案 {X}
→ 推荐理由：{引用具体上下文说明为什么推荐}
→ 请确认或选择其他方案。
```

## 阶段转换条件
以下条件满足后进入 Phase 3：
1. 架构模式已确定。
2. 服务端和客户端技术栈已确定。
3. 数据库选型已确定。
4. API 风格已确定。
5. 其他决策项可标注为"待定"在后续阶段补充，但核心 4 项不可跳过。
