# Phase 4 — 模块拆分（Modules）

## 阶段目标
基于 Phase 1 的功能清单和 Phase 2 的技术决策，识别核心模块，逐个引导完成模块级 Spec。

## 模块拆分流程

### 4.1 模块识别
**问题**：系统应该拆分为哪些核心模块？

引导要点：
- 从 P0/P1 功能清单出发，按业务领域聚合功能。
- 每个模块应有内聚的业务职责和清晰的边界。
- 遵循单一职责原则：一个模块只解决一类问题。
- 模块粒度建议：5-15 个核心模块（过多则过度拆分，过少则耦合过高）。

输出格式：
```
### 模块清单
| 模块 | 职责 | 包含的功能 | 重要程度 |
|------|------|-----------|---------|
| 用户模块 | 用户注册/登录/资料管理 | 注册、登录、找回密码、个人中心 | 核心 |
| ... | ... | ... | ... |
```

### 4.2 依赖关系图
**问题**：模块之间的依赖关系是什么？

引导要点：
- 识别模块间的调用方向（A → B 表示 A 依赖 B）。
- 检查是否存在循环依赖（必须消除）。
- 标注依赖强度（强依赖 / 弱依赖 / 事件驱动）。
- 确定模块 Spec 的编写顺序（被依赖的模块优先）。

输出格式：
```
### 模块依赖图
{文字描述依赖关系}

设计顺序（被依赖的优先）：
1. {基础模块}
2. {核心业务模块}
3. {上层应用模块}
```

### 4.3 逐模块 Spec 引导
对每个模块，按以下 11 个维度逐一引导：

---

#### 维度 1：模块职责与边界
**问题**：这个模块负责什么？不负责什么？

引导要点：
- 用一句话定义模块的核心职责。
- 明确列出"不负责"的事项，避免职责蔓延。
- 与相邻模块的边界划分依据。

#### 维度 2：数据结构
**问题**：这个模块的核心数据模型是什么？

引导要点：
- **实体（Entity）**：有唯一 ID、有生命周期的业务对象。
- **值对象（Value Object）**：无独立 ID、描述属性的对象。
- **DTO（Data Transfer Object）**：跨层/跨模块传输的数据结构。
- 每个字段：名称、类型、约束（非空/唯一/默认值/长度限制）。
- 实体间关系：一对一/一对多/多对多。
- 索引设计建议。

输出格式：
```
### 数据结构

#### 实体：{实体名}
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | bigint | PK, 自增 | 主键 |
| name | varchar(100) | NOT NULL | 名称 |
| status | tinyint | NOT NULL, 默认 0 | 状态枚举 |
| created_at | datetime | NOT NULL | 创建时间 |
| updated_at | datetime | NOT NULL | 更新时间 |

索引：
- idx_name(name) — 按名称查询
- idx_status_created(status, created_at) — 按状态+时间筛选

关系：
- {实体 A} 1:N {实体 B}（通过 {外键字段} 关联）
```

#### 维度 3：API 接口
**问题**：这个模块对外暴露哪些接口？

引导要点：
- 每个接口：HTTP Method、路由路径、请求参数、响应结构、错误码。
- 认证要求：哪些接口需要登录？哪些是公开的？
- 幂等性要求：哪些接口需要幂等？幂等键是什么？
- 分页策略：列表接口的分页方式（游标/偏移量）。
- 版本策略：API 版本管理方式。

输出格式：
```
### API 接口

#### {接口名称}
- 路由：{METHOD} {path}
- 认证：{需要/不需要}
- 幂等：{是/否}，幂等键：{字段}
- 请求参数：
  | 参数 | 位置 | 类型 | 必填 | 说明 |
  |------|------|------|------|------|
  | id | path | int | 是 | 资源 ID |
  | name | body | string | 是 | 名称 |
- 响应结构：
  | 字段 | 类型 | 说明 |
  |------|------|------|
  | code | int | 业务状态码 |
  | data | object | 响应数据 |
- 错误码：
  | 错误码 | 说明 | HTTP Status |
  |--------|------|------------|
  | 40001 | 参数校验失败 | 400 |
  | 40401 | 资源不存在 | 404 |
```

#### 维度 4：业务规则
**问题**：这个模块有哪些核心业务规则？

引导要点：
- **计算规则**：金额计算、折扣逻辑、排序算法。
- **校验规则**：字段合法性、业务可行性校验。
- **触发规则**：什么条件下触发什么动作（如"库存为 0 时自动下架"）。
- **约束规则**：业务层面的限制条件（如"每人每天限购 5 件"）。
- 规则的优先级：多个规则冲突时以哪个为准。

输出格式：
```
### 业务规则

| 规则编号 | 规则名称 | 规则描述 | 触发条件 | 执行动作 | 优先级 |
|---------|---------|---------|---------|---------|--------|
| BR-001 | 库存下架 | 库存为 0 时自动下架 | stock = 0 | 设 status = offline | P0 |
| BR-002 | 每日限购 | 每人每天限购 5 件 | 用户当日已购 >= 5 | 拒绝下单 | P0 |
```

#### 维度 5：边界条件
**问题**：哪些极端情况需要特殊处理？

引导要点：
- **极值边界**：最大/最小值、超长字符串、超大文件。
- **空值边界**：字段为空/null/空字符串时的行为。
- **并发边界**：同时操作同一资源时的处理（乐观锁/悲观锁/队列）。
- **幂等边界**：重复提交时的行为。
- **时间边界**：跨时区、夏令时、闰秒、零点切换。
- **容量边界**：列表最大长度、批量操作最大数量。

输出格式：
```
### 边界条件

| 场景 | 输入 | 期望行为 | 处理方案 |
|------|------|---------|---------|
| 名称超长 | name > 100 字符 | 返回参数校验错误 | 前端+后端双重校验 |
| 并发扣减库存 | 多用户同时下单 | 不超卖 | 乐观锁 + 库存预扣 |
| 重复提交订单 | 同一幂等键重复请求 | 返回上次结果 | 幂等键去重 |
```

#### 维度 6：状态机
**问题**：核心实体有哪些状态？如何流转？

引导要点：
- 列出所有合法状态。
- 定义合法的状态转换路径（哪个状态可以转到哪个状态）。
- 每次转换的触发条件（谁可以触发、什么条件下触发）。
- 不可逆状态标注（如"已删除"不可恢复）。
- 异常状态的处理（超时/失败后的状态回退策略）。

输出格式：
```
### 状态机：{实体名}

状态列表：
| 状态 | 编码 | 说明 | 是否终态 |
|------|------|------|---------|
| 待审核 | 0 | 初始状态 | 否 |
| 已通过 | 1 | 审核通过 | 否 |
| 已拒绝 | 2 | 审核拒绝 | 是 |

状态转换：
| 当前状态 | 目标状态 | 触发条件 | 触发角色 | 可逆 |
|---------|---------|---------|---------|------|
| 待审核 | 已通过 | 管理员审核通过 | admin | 否 |
| 待审核 | 已拒绝 | 管理员审核拒绝 | admin | 否 |
```

#### 维度 7：权限模型
**问题**：这个模块的访问控制如何设计？

引导要点：
- 角色定义：哪些角色可以访问本模块？
- 权限矩阵：每个角色对每个资源/操作的权限（CRUD + 特殊操作）。
- 数据权限：同一接口不同角色看到不同范围的数据。
- 权限继承：角色间的继承关系。

输出格式：
```
### 权限模型

| 操作 | 超级管理员 | 管理员 | 普通用户 | 访客 |
|------|----------|--------|---------|------|
| 创建 | ✅ | ✅ | ✅ | ❌ |
| 查看（全部） | ✅ | ✅ | ❌ | ❌ |
| 查看（自己的） | ✅ | ✅ | ✅ | ❌ |
| 编辑 | ✅ | ✅ | 仅自己的 | ❌ |
| 删除 | ✅ | ❌ | ❌ | ❌ |
```

#### 维度 8：事件与消息
**问题**：这个模块会发布或消费哪些事件？

引导要点：
- **发布的事件**：本模块状态变化时发出什么事件？谁会关心？
- **消费的事件**：本模块需要监听哪些外部事件？收到后做什么？
- 事件格式：事件名、payload 结构、幂等处理。
- 事件可靠性：是否需要保证至少投递一次？

输出格式：
```
### 事件清单

#### 发布的事件
| 事件名 | 触发时机 | Payload | 消费方 |
|--------|---------|---------|--------|
| order.created | 订单创建成功 | {orderId, userId, amount} | 库存模块、通知模块 |

#### 消费的事件
| 事件名 | 来源模块 | 处理逻辑 | 幂等策略 |
|--------|---------|---------|---------|
| payment.confirmed | 支付模块 | 更新订单状态为已支付 | orderId 去重 |
```

#### 维度 9：错误处理
**问题**：这个模块如何处理错误和异常？

引导要点：
- **错误码设计**：业务错误码体系（模块前缀 + 错误分类 + 序号）。
- **降级策略**：依赖的外部服务不可用时如何降级。
- **重试策略**：哪些错误可以重试？重试次数和间隔。
- **补偿策略**：分布式事务场景的补偿机制。
- **用户提示**：错误码到用户可读消息的映射。

输出格式：
```
### 错误处理

#### 错误码体系
| 错误码 | 含义 | HTTP Status | 用户提示 | 是否可重试 |
|--------|------|------------|---------|-----------|
| 10001 | 用户名已存在 | 409 | 该用户名已被使用 | 否 |
| 10002 | 密码强度不足 | 400 | 密码需至少 8 位 | 否 |

#### 降级策略
| 依赖服务 | 降级方案 | 影响范围 |
|---------|---------|---------|
| 短信服务 | 切换备用通道 | 验证码发送延迟 |
```

#### 维度 10：模块依赖关系
**问题**：这个模块依赖哪些其他模块？被谁依赖？

引导要点：
- 调用方向：本模块调用了哪些模块的接口？
- 被调用：哪些模块调用了本模块的接口？
- 依赖强度：强依赖（不可降级）/ 弱依赖（可降级）。
- 循环依赖检查：是否存在 A→B→A 的循环调用？

输出格式：
```
### 模块依赖

#### 依赖的模块
| 模块 | 调用接口 | 依赖强度 | 降级方案 |
|------|---------|---------|---------|
| 用户模块 | 获取用户信息 | 强依赖 | 无（核心链路） |

#### 被依赖的模块
| 模块 | 调用接口 | 说明 |
|------|---------|------|
| 通知模块 | 查询订单详情 | 发送订单通知时调用 |
```

#### 维度 11：测试策略
**问题**：这个模块的关键测试点有哪些？

引导要点：
- **单元测试**：核心业务规则的测试用例。
- **集成测试**：模块间调用的关键路径。
- **边界测试**：覆盖维度 5 中定义的边界条件。
- **性能测试**：需要压测的接口和目标指标。
- **端到端测试**：关键业务流程的全链路测试。

#### 维度 12：迁移与兼容性
**问题**：这个模块未来如何演进？

引导要点：
- API 版本策略：新版本如何兼容旧客户端？
- 数据迁移：Schema 变更时的数据迁移方案。
- 功能开关：新功能如何灰度上线？
- 弃用策略：旧接口的下线时间表和通知机制。

## 阶段输出
完成所有模块的 Spec 后，输出每个模块的完整 Spec 文档（按上述 12 个维度结构化）。

## 阶段转换条件
以下条件满足后进入 Phase 5：
1. 所有核心模块已完成 Spec（至少覆盖数据结构、API 接口、业务规则、边界条件 4 个必填维度）。
2. 模块依赖关系无循环依赖。
3. 跨模块的 API 调用已在被调用方的 Spec 中定义。
