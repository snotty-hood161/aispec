# Phase 5 — PRD 汇总（Summary）

## 阶段目标
整合前四个阶段的所有产出，生成一份完整的、可执行的产品需求文档（PRD），并标注与 Design Agent 和 Spec Agent 的衔接点。

## 汇总工作流

### 5.1 内容整合
将 Phase 1-4 的输出整合为 PRD 的各章节：
- Phase 1 市场调研 → PRD 第 1-2 章（背景与市场分析）
- Phase 2 竞品分析 → PRD 第 3 章（竞品分析）
- Phase 3 需求定义 → PRD 第 4-7 章（用户画像、用户故事、功能清单、业务规则）
- Phase 4 路线图 → PRD 第 8-10 章（MVP、路线图、风险）

### 5.2 一致性检查
- 功能清单是否覆盖了所有 P0 用户故事。
- 验收标准是否可测试、无歧义。
- 差异化定位是否在 MVP 功能中有体现。
- 非功能性需求是否与功能需求兼容。
- 成功指标是否与功能清单对应。

### 5.3 衔接点标注
标注 PRD 中需要下游 Agent 处理的内容：

| 衔接目标 | PRD 中的内容 | 下游 Agent 的动作 |
|---------|-------------|-----------------|
| Design Agent | 用户画像、用户故事、功能清单 | 设计交互流程、页面原型、视觉风格 |
| Spec Agent | 功能清单、业务规则、非功能性需求 | 技术选型、架构设计、模块拆分 |
| QA Agent | 验收标准、非功能性需求 | 测试策略、测试用例 |

## 阶段输出

使用 `agents/product/templates/prd-template.md` 模板输出完整 PRD。

### 输出要求
1. 所有输出使用中文。
2. PRD 文件保存至项目的 `docs/` 目录。
3. 文件命名：`docs/prd-{项目名}-v{版本号}.md`。
4. 竞品分析报告可独立保存：`docs/competitive-analysis-{项目名}.md`。

## 阶段完成条件
1. PRD 文档结构完整，所有章节已填充。
2. 一致性检查通过。
3. 衔接点已标注。
4. 用户确认 PRD 内容。
