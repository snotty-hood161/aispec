# Phase 3 — 需求定义（Requirements）

## 阶段目标
基于市场调研和竞品分析的洞察，定义目标用户、核心场景、功能清单和验收标准。

## 引导问题清单

### 3.1 用户画像定义
**问题**：我们的产品面向哪些用户角色？

引导要点：
- 识别 2-4 个核心用户角色。
- 每个角色包含：身份描述、核心诉求、使用场景、技术水平、使用频率。
- 区分主要用户（直接使用产品）和次要用户（间接受益）。
- 基于 Phase 2 的竞品分析，明确我们聚焦的用户群体。

### 3.2 用户故事编写
**动作**：为每个核心场景编写用户故事。

格式：
```
作为 {角色}，
我希望 {功能/行为}，
以便 {业务价值/用户价值}。

验收标准：
- [ ] {标准1}
- [ ] {标准2}
- [ ] {标准3}
```

编写原则：
- 每个用户故事必须包含验收标准。
- 验收标准必须可测试、可验证。
- 避免技术实现细节，聚焦业务行为。
- 使用 INVEST 原则（独立、可协商、有价值、可估算、小型、可测试）。

### 3.3 功能清单与优先级
**问题**：核心功能有哪些？按什么标准排序优先级？

引导要点：
- 从用户故事中提取功能点。
- 按 P0/P1/P2 分级：
  - **P0 — 必须有**：MVP 必须包含，没有则产品无法使用。
  - **P1 — 应该有**：首版上线后第一个迭代补齐。
  - **P2 — 可以有**：未来规划，当前不阻塞。
- 标注功能间的依赖关系。
- 对照竞品分析，确保差异化功能被标记为高优先级。

### 3.4 非功能性需求
**问题**：产品在性能、安全、可用性等方面有什么要求？

引导要点：
- 性能要求（页面加载时间、并发用户数、响应延迟）。
- 安全要求（数据加密、认证方式、合规标准）。
- 可用性要求（SLA、容灾、降级策略）。
- 无障碍要求（WCAG 标准、国际化/多语言）。
- 兼容性要求（浏览器、设备、操作系统）。

### 3.5 业务规则
**问题**：有哪些核心业务规则需要明确？

引导要点：
- 权限模型（角色、权限粒度）。
- 数据规则（数据保留期、导出格式、隐私要求）。
- 计费/限额规则（如适用）。
- 审批/工作流规则（如适用）。
- 通知与消息规则。

## 阶段转换条件
以下条件满足后进入 Phase 4：
1. 至少定义 2 个用户角色。
2. P0 功能的用户故事已全部编写，含验收标准。
3. 功能优先级矩阵已完成。
